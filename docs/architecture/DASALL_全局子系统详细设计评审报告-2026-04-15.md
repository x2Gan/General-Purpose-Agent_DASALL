# DASALL 全局子系统详细设计评审报告

版本：v1.1
评审日期：2026-04-15
评审范围：DASALL 各子系统详细设计、contracts 冻结实施计划、boundary 治理说明
评审角色：C++ / Agent 架构评审

---

## 1. 评审目标

本报告用于从全局视角评审 DASALL 已形成的子系统详细设计，重点检查以下方面：

1. 子系统集成是否能闭合为单 Agent 主链路。
2. 子系统依赖方向、跨子系统能力边界是否稳定。
3. 子系统共享数据面是否足以支撑当前设计，是否需要继续演进 contracts。
4. 除职责分层之外，配置治理、恢复治理、可观测性、测试门、实施顺序是否成立。
5. 现有文档集是否仍有关键缺口，导致后续 Build 阶段出现系统级返工。

本次评审的核心判断不是“单个子系统是否写得完整”，而是“这些详细设计能否拼成一个可实现、可测试、可恢复、可演进的系统”。

### 1.1 本轮优化补充说明

1. 本轮在原始评审结论之上，补充了面向 Agent / 分布式系统 / 文档治理的行业实践调研，用于判断“哪些结论应收紧、哪些结论应加码、哪些结论只适合留在 module-local”。
2. 调研采用“抽取可迁移原则”的方式，而不是把某个云厂商或框架方案直接平移到 DASALL；目标是为 DASALL 的 C++ 单 Agent 主链给出更稳的工程落点。
3. 按本轮补充要求，本文明确跳过 multi_agent 子系统详细设计缺失问题，不把它作为本版优化评审的阻塞项和新增收敛项。

### 2.1 行业实践调研摘要

1. 合同与集成：成熟系统通常先定义最小 provider contract / consumer contract，再用 narrow integration test 与少量高价值 smoke Gate 守住主链，而不是先堆大而全 shared schema。
2. RAG / Knowledge：生产级 RAG 普遍保留 provenance、freshness、version、anchor 等证据元数据，而不是只保留纯文本切片。
3. Access / Security：鉴权应 fail-closed、每请求校验、服务端或网关侧执行，并把 actor / action / target / evidence 以一致格式沉淀到审计链。
4. 配置治理：生效配置应表现为 typed、versioned、可回滚的 snapshot，变更应像代码一样经过 staged rollout 与验证。
5. 恢复治理：重试、熔断、补偿、幂等性必须由掌握完整上下文的控制面裁定，建议层只给出策略候选，不直接持有执行权。
6. 文档治理：Docs as Code 的有效前提不是“文档进 Git”这么简单，而是文档也要接受事实校验、评审与自动化一致性检查。

## 3. 总体结论

### 3.1 一句话结论

DASALL 当前的全局架构方向基本正确，关键边界守得住，整体可判定为 architecture ready；但跨子系统接口面、共享数据投影面和集成 Gate 仍未闭合，系统整体仍处于 integration not ready / implementation largely not ready。

### 3.2 维度结论表

| 维度 | 结论 | 评估 |
|---|---|---|
| 控制面分层 | Runtime、Memory、LLM、Tools、Cognition、Access 的一阶边界总体正确 | 通过 |
| 依赖方向 | 大多数详细设计遵守 contracts 横切、下层不反向依赖上层、Runtime 唯一主控 | 通过 |
| 共享数据面 | Observation、ContextPacket、Checkpoint 等主对象骨架正确，但证据投影和 access sidecar 仍过薄 | 有条件通过 |
| contracts 治理 | 冻结策略整体克制，未出现大规模“把 module-local supporting objects 硬推进 shared layer”的问题 | 通过 |
| 配置与 Profile 治理 | RuntimePolicySnapshot 设计合理，但缺少系统级消费矩阵，存在多模块各自解释风险 | 有条件通过 |
| 恢复与安全治理 | ADR-007 的建议权/执行权分离正确，但运行时所需恢复上下文表还不够明确 | 有条件通过 |
| 可观测性与测试门 | 单模块设计普遍重视日志/指标/审计/集成测试，但跨模块 Gate 尚未成为一等资产 | 有条件通过 |
| 实施顺序可执行性 | 阶段化思路正确，但 runtime 阶段 J 仍受 memory / knowledge / tools 公共接口与 integration smoke 缺失阻塞 | 未通过 |

### 3.3 总体判断

1. 当前最不该做的事情，是在 runtime、cognition、tools、memory、knowledge 之间直接并行编码，而让接口、投影、集成 Gate 留到实现时临时协商。
2. 当前最该做的事情，不是扩张更多 shared contracts，而是先闭合单 Agent unary 主链的跨模块接口和数据投影。
3. 只有在 P0 级的接口闭口与投影闭口完成后，现有详细设计才适合进入更大规模的 Build 并行阶段。

## 4. 关键发现（按严重度排序）

### GLOB-01（P0）单 Agent 主链尚未形成可验证的集成闭环

**问题描述**

Runtime 详细设计已经把自身定义为唯一全局主控，但站在“runtime 是 consumer”的视角看，memory / knowledge / tools 面向它的最小 provider contract 仍未真正冻结，导致“架构上闭合”与“工程上可联调”之间仍有断层。

**证据**

1. [DASALL_runtime子系统详细设计.md](DASALL_runtime子系统详细设计.md) 在现状表中明确写出 memory / knowledge / tools 公共接口尚未落地。
2. [DASALL_runtime子系统详细设计.md](DASALL_runtime子系统详细设计.md) 在差距表中明确写出没有 runtime + cognition + llm + tools + memory 的真实联动路径。
3. 当前仓库中 llm/include 已存在，而 runtime/include、memory/include、knowledge/include、tools/include 仍未形成对等的主链公共接口面。

**行业实践对照**

1. Consumer-Driven Contracts 的核心不是先写大而全接口，而是由实际 consumer 提出最小义务面，再用自动化 provider verification 持续守住它。
2. Test Pyramid 与 narrow integration test 的成熟做法强调：大多数信心应来自边界级集成测试，只有极少数高价值 broad smoke 才应该被提升为 pipeline Gate。
3. CMake workflow preset 这类工程化实践表明，跨模块 configure -> build -> test 的关键闭环如果不固化为可执行工作流，最终会退化成口头 Gate。

**系统影响**

1. runtime 阶段 J 的“单 Agent 生产级闭环”无法真正落到代码和测试。
2. 各子系统虽然都写了 Design -> Build 映射，但系统层没有一条可以验证 request -> decide -> execute -> observe -> recover -> respond 的真实 smoke。
3. 后续实现极易退化成 runtime 直接 include 相邻模块实现细节，破坏当前已经写清的依赖方向。

**优化后的判断**

1. DASALL 的问题不在于 Runtime 主控定义错误，而在于 Runtime 作为 consumer 的 contract surface 仍未形成可执行、可验证的最小义务集。
2. 当前真正需要收紧的是 unary mainline 的接口闭口与 Gate，而不是继续扩 streaming、更多 worker lane 或更多 supporting objects。

**优化后的建议动作**

1. 用“runtime 作为 consumer”的方式冻结 memory / knowledge / tools 最小 provider contract：请求对象、返回对象、失败语义、超时语义、degrade 语义一起定义，不只定义 happy path 方法签名。
2. 为每个 provider contract 配一组 narrow integration tests；系统层只保留一条 single-agent unary smoke，覆盖 request -> decide -> execute -> observe -> recover -> respond。
3. 将 unary smoke 固化为可执行 Gate，例如 CMake/CTest label 或 workflow preset，而不是文档口头约定。

**contracts 判断**

不需要立即演进 shared contracts；优先补 module public interface、failure taxonomy 与可执行集成 Gate。

### GLOB-02（P0）Knowledge 到 Memory / Cognition 的共享证据面过薄

**问题描述**

Knowledge 明确产出 EvidenceBundle / EvidenceSlice 这类结构化证据，但跨模块共享面当前仅剩 ContextPacket.retrieval_evidence 的字符串数组。这个投影对“来源、可信度、新鲜度、引用锚点、版本”的保真度不足，已经不能完全支撑 memory 和 cognition 详细设计中提出的上下文充分性、证据回链和反思可解释性要求。

**证据**

1. [DASALL_knowledge子系统详细设计.md](DASALL_knowledge子系统详细设计.md) 明确写出当前共享证据投影面只有 retrieval_evidence: vector<string>。
2. [DASALL_knowledge子系统详细设计.md](DASALL_knowledge子系统详细设计.md) 同时又明确写出 EvidenceSlice / EvidenceBundle 与 ContextPacket.retrieval_evidence 的投影逻辑仍缺失。
3. [DASALL_memory子系统详细设计.md](DASALL_memory子系统详细设计.md) 要求 ContextOrchestrator / CandidateCollector 消费 external_evidence。
4. [DASALL_cognition子系统详细设计.md](DASALL_cognition子系统详细设计.md) 已经要求 ContextSufficiencySignal、Observation freshness、relevant_observation_refs、evidence_refs_delta 等更细粒度语义。

**行业实践对照**

1. 生产级 RAG 方案通常保留 provenance、freshness、document version、chunk anchor 等元数据，用于 rerank、citation、fact check 与 root-cause analysis，而不是只保留纯文本片段。
2. 服务演进实践强调 additive / optional 的兼容扩展，以及“just enough validation”：只共享 downstream 确实消费的最小义务面，而不把 provider 内部 supporting types 整体外推。

**系统影响**

1. Knowledge -> Runtime -> Memory 的 handoff 会丢失结构化证据语义。
2. cognition 无法稳定地对检索证据做 freshness、trust、explainability 级判断。
3. 如果继续只用字符串数组，后续要么在 runtime / memory / cognition 中各自引入私有补丁字段，要么返工 contracts。

**优化后的判断**

1. 当前缺的不是把 EvidenceSlice / EvidenceBundle 整体推进 shared contracts，而是为跨模块 handoff 增加一层更薄的 provenance projection。
2. 这也是本轮评审里唯一值得推进到 contracts 的新增面，因为它直接影响 unary 主链的数据保真度与可解释性。

**优化后的建议动作**

1. 在 contracts 中增加“最小结构化证据投影”，但保持 optional + additive，不破坏现有 retrieval_evidence 文本投影兼容路径。
2. 保持 KnowledgeQuery、EvidenceSlice、EvidenceBundle 继续为 module-local supporting types，不把检索内部策略对象外溢到 shared layer。
3. 为 Knowledge -> Runtime -> Memory / Cognition 增加 citation preservation regression cases，验证 evidence_ref、freshness、anchor 在主链上不丢失。

**建议的最小新增面**

建议优先评估新增一个轻量结构，例如 `RetrievalEvidenceRef` 或等价对象，仅承载：

1. evidence_ref
2. source_ref
3. source_kind
4. summary_text
5. trust_level
6. collected_at 或 freshness
7. optional: anchor_locator
8. optional: document_version

以下字段继续保留 module-local：ranking score、full chunk text、query rewrite trace、embedding / ANN 检索内部统计。

**contracts 判断**

这是本次评审中最明确的 contracts 演进候选项。

### GLOB-03（P0）Access 到 Runtime 的安全 sidecar 尚未闭口

**问题描述**

Access 的总体边界设计是正确的：主体身份、授权证明、异步回执、流式状态都不应进入 shared contracts，而应留在 module-local sidecar。但目前 access-runtime bridge 尚未真正落盘，导致“接入侧认证鉴权结果如何稳定传入 runtime 并映射到审计 actor”这条链还没有工程化定义。

**证据**

1. [DASALL_access子系统详细设计.md](DASALL_access子系统详细设计.md) 明确写出 contracts/include/boundary/IdentityMetadata.h 不是 access 主体身份契约。
2. [DASALL_access子系统详细设计.md](DASALL_access子系统详细设计.md) 明确要求主体上下文、策略证明和发布上下文留在 module-local sidecar。
3. [DASALL_access子系统详细设计.md](DASALL_access子系统详细设计.md) 也明确写出 Runtime bridge 缺失。
4. [DASALL_infrastructure子系统详细设计.md](DASALL_infrastructure子系统详细设计.md) 对 AuditEvent、override actor、审计 evidence_ref 都要求可追踪和可审计。

**行业实践对照**

1. 安全治理的通行实践是 deny-by-default、每请求鉴权、服务端或网关侧执行，而不是把原始外部上下文直接下沉到核心控制面再二次解释。
2. Gateway Routing / BFF 的经验表明，鉴权、路由、监控这类 cross-cutting concern 应停留在边界层，并向内提供规范化 proof，而不是把边界原始对象泄漏到内部主链。
3. 复杂场景更适合 ABAC / ReBAC 风格的主体属性与资源属性组合，而不是把粗粒度 RBAC 角色直接绑死在 shared contracts。

**系统影响**

1. access 的 fail-closed、认证、授权虽然在设计上成立，但还没有一条稳定的 runtime handoff。
2. 审计 actor、request_id、override issued_by 等横切字段容易在 access、runtime、infra 三处各自拼装，产生漂移。
3. 后续 async receipt、重放、流式重连会因为缺少同一 sidecar 语义而重复造对象。

**优化后的判断**

1. 该问题的本质不是 contracts 不够大，而是 access 还没有把“边界层已裁定的最小事实”压缩成 runtime 可消费的 sidecar proof。
2. 只要 handoff 对象够稳定、够可审计，主体身份与流式细节仍应继续停留在 access 侧，不应污染 shared contracts。

**优化后的建议动作**

1. 在 access/include 中定义 RuntimeDispatchRequest、AccessDecisionProof、AccessSubjectContext 等 module public types，但限定它们只服务 access -> runtime handoff。
2. AccessDecisionProof 至少承载 decision_id、subject attributes、resource/action tuple、policy_version、issued_at、expires_at、request_id / correlation_id。
3. 在 runtime 入口面明确接收“AgentRequest + access sidecar”的组合，而不是直接吞 access 私有对象或原始外部 claims。
4. 将 actor / action / target / outcome / evidence_ref 的 access-runtime-infra 映射写成一份表，并补 deny / allow / expired proof / replay proof 的负向测试。

**contracts 判断**

不建议推进 shared contracts。该问题应通过 access/runtime/infra 三者的 module public interface 和审计映射表解决。

### GLOB-04（P1）RuntimePolicySnapshot 缺少系统级消费矩阵，存在多模块各自解释风险

**问题描述**

profiles 的双平面设计正确，RuntimePolicySnapshot 也已经覆盖 runtime_budget、model_profile、token_budget_policy、prompt_policy、capability_cache_policy、degrade_policy、timeout_policy、execution_policy、ops_policy。但当前仍缺少一份系统级“谁消费什么键、谁拥有该键的语义解释权”的矩阵。

**证据**

1. [DASALL_profiles模块详细设计.md](DASALL_profiles模块详细设计.md) 已定义 RuntimePolicySnapshot 和 schema_version: 1 的固定逻辑域。
2. [DASALL_tools子系统详细设计.md](DASALL_tools子系统详细设计.md) 使用 max_tool_calls、tool_visibility_rules、allowed_tool_domains、timeout_policy.*。
3. [DASALL_llm子系统详细设计.md](DASALL_llm子系统详细设计.md) 使用 model_profile、prompt_policy、timeout_policy、degrade_policy。
4. [DASALL_knowledge子系统详细设计.md](DASALL_knowledge子系统详细设计.md) 使用 enabled_modules.knowledge、memory_vector、token_budget_policy、capability_cache_policy。
5. [DASALL_memory子系统详细设计.md](DASALL_memory子系统详细设计.md) 也依赖 token_budget_hint、latest_observation_digest_summary、external_evidence 的预算裁剪路径。

**行业实践对照**

1. 外部配置存储的成熟做法是 typed、versioned、可缓存、可回滚的 effective snapshot，而不是让每个模块直接解释原始配置源。
2. 多租户与渐进发布实践要求配置变更像代码一样经过 staged rollout、测试与回滚，不应通过多个模块各自热解释来碰运气。
3. CMake Presets 这类项目级 shared preset 与开发者本地 override 分离的做法，也说明了“共享治理基线”和“本地临时覆写”必须分层表达。

**系统影响**

1. 同一个 profile 键容易被多个模块各自本地解释，造成“文档兼容、运行漂移”。
2. 后续 deployment_override / runtime_override 一旦引入更多热更新，问题会放大。
3. profile 作为全局治理基线，如果没有消费矩阵，就很难证明真正的单一真相源。

**优化后的判断**

1. RuntimePolicySnapshot 的方向是对的，但它现在更像“字段集合”，还不是“带 owner 与 override 规则的生效配置契约”。
2. DASALL 真正缺的不是更多 profile key，而是一份把语义 owner、消费范围、热更新行为写清的系统级矩阵。

**优化后的建议动作**

1. 将 RuntimePolicySnapshot 明确为“编译后的生效快照”，按 request、session 或 orchestration cycle 确定其稳定边界，避免模块在一次主链执行中读取到不一致视图。
2. 补一份消费矩阵，至少包含：key、semantic owner、primary consumer、secondary consumer、override source、hot-reload 行为、fallback、validation rule。
3. 对 token_budget_policy、capability_cache_policy、timeout_policy 这类多模块共享键，补统一语义说明与示例，明确“谁可读、谁可解释、谁不可重定义”。

**contracts 判断**

不需要演进 contracts；需要补系统级 SSOT，并把 profile 变更纳入 staged Gate。

### GLOB-05（P1）ADR-007 的建议权与执行权虽然分离正确，但恢复上下文仍不够显式

**问题描述**

目前 cognition 和 runtime 都正确承认：ReflectionEngine 只给建议，RecoveryManager 持有最终恢复执行权。但“cognition 看得到什么，runtime 额外组合什么，哪些信息绝不能回流给 cognition”还没有形成一份足够明确的恢复上下文边界表。

**证据**

1. [DASALL_cognition子系统详细设计.md](DASALL_cognition子系统详细设计.md) 将 retry_step、replan、abort_safe 明确定义为建议。
2. [DASALL_runtime子系统详细设计.md](DASALL_runtime子系统详细设计.md) 明确 runtime 需要将 ReflectionDecision 与预算、幂等性、副作用、熔断状态组合后再裁定。
3. [DASALL_cognition子系统详细设计.md](DASALL_cognition子系统详细设计.md) 同时禁止 cognition 直接读取 retry counters、checkpoint blob、backoff。

**行业实践对照**

1. 重试模式的通行做法是：只有掌握完整上下文的一层才能决定是否 retry；否则容易出现重复重试、延迟叠加和副作用放大。
2. 补偿事务 / saga 的实践强调：恢复动作需要 durable undo info、idempotent command、timeout 与人工介入路径，这些都属于执行控制面，而不是建议层。
3. 工作流引擎类实践会把 timeout、history、retries、observability、runtime safeguard 放在 control plane，而不是让 planner / advisor 直接触达执行细节。

**系统影响**

1. “何时允许 replan / retry，何时必须 stop / degrade” 的测试用例难以统一建模。
2. 若实现阶段临时在 cognition 中补恢复上下文，会直接侵蚀 ADR-007。
3. 若 runtime 不显式声明其额外组合的信息，后续恢复拒绝原因会变得不可解释。

**优化后的判断**

1. DASALL 的边界原则是对的，但恢复决策还缺一份“控制面最小判定上下文模型”。
2. 在没有这份上下文表之前，任何实现都可能因为临时补字段而让建议层与执行层重新耦合。

**优化后的建议动作**

1. 补一份 Recovery Context 表，至少区分：cognition 可见事实、runtime 独占事实、共享输入、禁止回流项、幂等性分类、补偿句柄、熔断状态、deadline / retry budget。
2. 在 runtime 阶段 J 的 Gate 中加入 failure injection 场景，至少覆盖：连续失败、长期不可用、部分副作用已发生、重复回放、重规划上限触发。
3. 保持 ReflectionDecision、RecoveryRequest、RecoveryOutcome 三对象不扩张控制字段；执行裁定所需上下文继续停留在 runtime 内部或 runtime-facing interface。
4. 明确“补偿失败后的人工介入路径”属于 runtime / ops 责任，而不是让 cognition 负责补救执行后果。

**contracts 判断**

短期不需要扩 contracts；优先补 runtime-cognition 边界表、failure injection Gate 与 side-effect taxonomy。

### GLOB-06（P2）当前部分“现状与缺口”章节已与仓库事实产生漂移

**问题描述**

个别详细设计中的“当前仓库现状”已经落后于实际目录状态，最典型的是 llm 文档仍写着 llm/include 不存在，而仓库已经存在 llm/include/ILLMManager.h、ILLMAdapter.h 等公共头，runtime 详设也已把它们视为现成依赖。

**证据**

1. [DASALL_llm子系统详细设计.md](DASALL_llm子系统详细设计.md) 现状段仍写 llm/include 目录不存在。
2. [DASALL_runtime子系统详细设计.md](DASALL_runtime子系统详细设计.md) 已明确把 llm/include/ILLMManager.h、ILLMAdapter.h 视作已存在的公共接口。

**行业实践对照**

1. Docs as Code 的成熟实践并不止于“文档进版本库”，而是要求文档同样进入 code review、自动化检查与事实校验链。
2. 持续文档化的经验表明，凡是带“当前状态 / 当前目录 / 当前接口”判断的章节，都应绑定证据时间点或自动化事实检查，否则极易变成滞后快照。

**系统影响**

1. 设计评审和实施排期会对“哪些是 blocker、哪些已存在”产生误判。
2. 容易出现已经完成的工程基线在文档里仍被当成未完成，影响阶段判定。

**优化后的判断**

1. 这不是单纯的措辞问题，而是会反向污染系统级优先级判断的问题。
2. 既然 DASALL 依赖 Design -> Build 双轨推进，文档事实漂移就必须被视为工程风险，而不是写作细节。

**优化后的建议动作**

1. 把“现状与缺口”纳入 doc-to-repo 一致性检查，优先校验 include 目录、公共头、CMake target、关键 preset 与测试 Gate。
2. 对 architecture ready / implementation not ready 的判定增加“目录事实校验”，必要时在章节中标注 evidence date 或 repo snapshot。
3. 评审发现的漂移项应落回 TODO / worklog，而不是只停留在 review report 中。

**contracts 判断**

不涉及 contracts；属于文档治理问题，但会直接影响系统收敛顺序。

## 5. 对 contracts 演进的判断

### 5.1 建议推进的演进项

| 项目 | 判断 | 行业实践支撑 | 建议策略 |
|---|---|---|---|
| 最小结构化证据投影 | 建议推进 | 生产级 RAG 需要 provenance / freshness / citation anchor；兼容演进应优先 additive + optional | 在 ContextPacket 中新增轻量结构化 evidence refs，保留现有 retrieval_evidence 文本投影兼容路径 |

### 5.2 明确不建议推进 shared contracts 的项

| 项目 | 结论 | 原因 |
|---|---|---|
| ActionDecision / PlanGraph / ReplanResult | 不推进 | 仍属于 cognition module-local supporting types；行业实践倾向只共享 consumer 真正依赖的最小义务面 |
| BeliefUpdateHint | 不推进 | 目前是 cognition -> runtime / memory 的写回提示，不应过早冻结为共享补丁语义 |
| ToolRouteDecision / CapabilitySnapshot / SkillSpec | 不推进 | tools 详细设计已明确这些对象保持 module-local，过早共享会放大 ABI 返工 |
| SubjectIdentity / AccessDecisionProof / AsyncTaskReceipt / StreamSubscription | 不推进 | 这些都是 access module-local sidecar 或发布面对象；成熟做法通常把它们固定在边界层，而不是污染主链 shared contracts |
| IExecutionService / IDataService supporting objects | 暂不推进 | 接口本身仍在 services/include 收敛，supporting objects 尚未成熟，不宜进入 contracts |
| MemoryContextRequest / MemoryWritebackRequest | 不推进 | 这是 runtime-facing module public interface，而不是跨模块共享数据语言 |

### 5.3 最小结构化证据投影的补充建议

建议最小对象只承载 downstream 确实消费的共享字段：

1. evidence_ref
2. source_ref
3. source_kind
4. summary_text
5. trust_level
6. collected_at 或 freshness
7. optional: anchor_locator
8. optional: document_version

建议继续留在 module-local 的字段：

1. full chunk text
2. ranking score / ANN 细节
3. query rewrite / router trace
4. provider-specific retrieval rationale

### 5.4 对 Observation / ObservationDigest 的判断

本次评审不建议新增一套平行“工具结果投影共享对象”。

原因如下：

1. [DASALL_contracts冻结实施计划.md](../plans/DASALL_contracts冻结实施计划.md) 已经明确 Observation 是 Tool、Knowledge、Human、Worker 输出的统一折叠对象，ObservationDigest 是其推理消费侧摘要。
2. 当前真正缺的是 [DASALL_tools子系统详细设计.md](DASALL_tools子系统详细设计.md) 中 ResultProjector 的落地与测试 Gate，而不是再造一层新的 shared result object。
3. 按测试金字塔思路，更高价值的动作是补投影链路测试，而不是增加又一层 shared type。

因此，对工具链路的正确动作应该是：

1. 完成 ToolResult -> Observation -> ObservationDigest 的显式投影设计与实现。
2. 对 Observation 与 ObservationDigest 做跨模块集成测试。
3. 不额外扩张 shared contracts 面。

## 6. 其它有价值的评估维度

### 6.1 配置治理维度

优点：

1. profiles 双平面机制方向正确。
2. 多数模块都在克制地“消费生效策略视图”，没有直接回写 profile schema。

缺口：

1. 缺少统一的 RuntimePolicySnapshot 消费矩阵。
2. deployment_override / runtime_override 对跨模块影响缺少 staged rollout 与统一系统级验证。

### 6.2 可观测性与审计维度

优点：

1. infra、access、tools、runtime、llm 都高度重视日志、指标、trace、audit。
2. fail-closed 和审计 actor / evidence_ref 思路是正确的。

缺口：

1. 仍缺跨模块指标白名单和事件命名矩阵。
2. 仍缺系统级“关键事件必留痕”清单，例如 access deny、runtime recovery reject、knowledge degrade、tool compensation hint emitted。

### 6.3 实施顺序维度

优点：

1. 文档整体尊重 contracts 优先、模块边界优先、shared object 克制推进的策略。
2. runtime 已明确 single-agent unary 应先于 streaming 与更多 shared supporting objects。

缺口：

1. 尚未形成一套真正的跨模块阶段 Gate。
2. Gate 还没有绑定到仓库中的可执行预设、测试标签与工作流资产。

### 6.4 文档治理维度

优点：

1. 各子系统详设结构普遍统一，Design -> Build 映射意识较强。
2. architecture ready / implementation not ready 的表达基本到位。

缺口：

1. “现状与缺口”章节与实际仓库目录存在漂移风险。
2. 仍缺 doc-to-repo 自动校验与 review -> todo / worklog 的证据闭环。

1. 不是某个具体算法没设计，而是跨模块数据投影谁负责还没系统级写清。

3. 不是 runtime 不够复杂，而是 runtime 虽被定义为主控，但主链依赖端口和集成 Gate 还没全部落地。

## 8. 建议的收敛顺序

### 8.1 P0：先闭合单 Agent unary 主链

1. 补 memory / knowledge / tools 的 runtime-facing module public interface，并补 failure taxonomy。
2. 补 access-runtime sidecar 与最小 handoff proof。
3. 补结构化证据投影的 contracts 演进方案，并加 citation preservation regression cases。
4. 建立 runtime + cognition + llm + tools + memory 的 unary integration smoke，并把它固化为可执行 Gate。

### 8.2 P1：再闭合治理与恢复边界

1. 补 RuntimePolicySnapshot 消费矩阵。
2. 补 Recovery Context 边界表与 failure injection Gate。
3. 补 Access-Runtime-Infra 审计映射表。

### 8.3 P2：最后收敛治理质量与维护性

1. 补跨模块指标白名单和关键事件矩阵。
2. 补文档现状一致性检查。
3. 持续把 architecture ready / implementation not ready 的判定与仓库事实绑定。

## 9. 最终结论

1. DASALL 当前的子系统详细设计在“边界治理”上是强于多数同类 Agent 工程的，尤其是 Runtime 主控权、Memory 上下文权、LLM Prompt 渲染权、Recovery 准入权的分离已经基本立住。
2. 当前最大的系统级风险不是职责错配，而是跨模块接口与投影面过薄，导致 integration readiness 落后于 architecture readiness。
3. 当前最值得推进的 contracts 演进只有一个：为 Knowledge -> Memory / Cognition 提供更稳的最小结构化证据共享面。
4. 当前最不该推进的，是把 ActionDecision、PlanGraph、BeliefUpdateHint、ToolRouteDecision、Access sidecar 等 supporting objects 过早写入 shared contracts。
5. 从行业实践看，DASALL 现在最需要的不是更多概念对象，而是更小、更硬的 consumer contract、snapshot contract 和 recovery context contract。
6. 按本轮优化要求，multi_agent 相关缺口不纳入当前阻塞项；当前收敛焦点应严格锁定 single-agent unary 主链。
