## 1. 执行摘要

本报告用于回答一个更具体的问题：截至 2026-05-06，DASALL 的各子系统是否已经从“详细设计上可以拼起来”推进到“代码与测试上已形成可验证的单 Agent 集成闭环”。

本轮评审不再重复判断架构方向是否合理，而是把重点放在以下五个维度：公共数据结构、对外接口、业务链路、代码实现与设计一致性，以及配置、恢复、可观测性、测试门等系统级治理维度。

核心结论如下：

1. DASALL 的一阶架构边界总体成立，ADR-006、ADR-007、ADR-008 对 Memory、Cognition、Runtime 的权责划分基本守住。
2. access、cli、daemon、platform_linux、tools、capability services 的代表性集成面已经较为扎实，能够证明入口链与工具执行链并非空设计。
3. runtime 当前已经不是骨架实现，真实主链会调用 memory、cognition、tools 并生成 AgentResult；但默认 live unary 主链的 focused gate 仍为红灯，当前不能宣称“系统主链已集成就绪”。
4. knowledge 与 llm 子系统本身并非未实现，但它们目前仍是 runtime 的可插拔接缝，而不是默认 live unary 闭环的必需依赖；这使“子系统可用”与“系统主链闭合”之间仍有落差。
5. 当前最关键的系统级缺口集中在三处：runtime/cognition 的最终响应合同漂移、knowledge 结构化证据在主链上的扁平化损失、infra diagnostics retained snapshot 的回读回归。

综合判断：DASALL 当前处于 architecture ready、subsystem mostly ready、integration partially ready、production not ready 的阶段。后续工作不应继续泛化为“全模块并行实现”，而应围绕单 Agent unary 主链与系统级 Gate 做定点收敛。

### 2.3 评估口径

本报告使用以下状态语义：

1. 绿色：接口、链路与代表性测试基本一致，可以作为当前阶段的稳定基座继续推进。
2. 黄色：设计方向与主要实现成立，但仍存在系统级缺口或关键 gate 未完全闭合。
3. 红色：当前已出现主链回归或关键验证失败，不能作为“已集成就绪”的结论基础。

## 3. 总体判断

### 3.1 维度结论表

| 维度 | 当前判断 | 结论 |
|---|---|---|
| 公共数据结构 | 有条件通过 | contracts 主骨架稳定，但 knowledge 到 memory/cognition 的证据投影仍过薄，结构化 provenance 在主链上被压缩为字符串数组 |
| 对外接口 | 有条件通过 | access/runtime、tools/services、cli/daemon 等对外接口基本成形，但 runtime 对 knowledge/llm 的准入条件与产品语义尚未统一 |
| 业务链路 | 未通过 | cli -> daemon -> access -> runtime 与 tools -> services 两条代表性链路可跑，但默认 live unary 主链仍被 runtime/cognition 响应合同漂移阻断 |
| 代码实现与设计一致性 | 有条件通过 | 多数子系统已从文档推进到真实实现；但个别“现状与缺口”章节仍落后于仓库事实，且 response contract 与 diagnostics snapshot 行为出现回归 |
| 配置/恢复治理 | 有条件通过 | ADR 边界总体成立，但 RuntimePolicySnapshot 的系统消费矩阵和恢复上下文边界仍需作为 SSOT 强化 |
| 可观测性与测试门 | 有条件通过 | access、tools、knowledge、llm、platform_linux 的 smoke 已能提供正向证据；infra diagnostics 和 runtime 真集成 gate 仍为红灯 |

### 3.2 总体结论

1. DASALL 现在最需要修复的是“系统主链的少数红灯”，而不是继续泛化设计对象或扩张 shared contracts。
2. 如果按成熟工程的标准判断，当前可以宣布“若干子系统已具备独立可运行证据”，但还不能宣布“单 Agent 默认主链已集成闭合”。
3. 因此，本阶段最合理的推进策略是围绕 runtime unary 主链、structured evidence projection、infra diagnostics retained snapshot 三个中心点收敛，而不是无差别铺开更多功能面。

## 4. 关键发现

### GINT-01（P0）Runtime 主链已真实存在，但默认 live unary gate 仍为红灯

**问题描述**

runtime 当前已经不是 placeholder。真实主链会准备上下文、调用 cognition 决策/反思、执行 tools、投影 Observation，并最终构建 AgentResult。但这条真实主链的 focused integration gate 仍然失败，表明当前问题是“主链行为漂移”，不是“尚未接线”。

**证据**

1. [../../runtime/src/AgentOrchestrator.cpp](../../runtime/src/AgentOrchestrator.cpp) 中存在 external_evidence 投影、memory prepare_context、tool_manager invoke、response_builder build 与最终 AgentResult 赋值等完整主链步骤。
2. [../../runtime/src/AgentFacade.cpp](../../runtime/src/AgentFacade.cpp) 真实调用 orchestrator->run_once，而非仅返回固定骨架结果。
3. [../../tests/integration/agent_loop/RuntimeUnaryIntegrationTest.cpp](../../tests/integration/agent_loop/RuntimeUnaryIntegrationTest.cpp) 与 [../../tests/integration/cognition/CognitionRuntimeIntegrationTest.cpp](../../tests/integration/cognition/CognitionRuntimeIntegrationTest.cpp) 的最终 response_text / Completed 状态断言当前失败。
4. [../../cognition/src/response/ResponseBuilder.cpp](../../cognition/src/response/ResponseBuilder.cpp) 当前对 observation projection 和 llm bridge unavailable 的回退组合，会直接影响最终 response_text 语义。

**影响**

1. 无法把 runtime 宣布为“真集成 ready”的主控子系统。
2. runtime 与 cognition 虽然已经连通，但最终结果合同没有锁稳，后续所有更上层入口链都会承受语义漂移。
3. 这类问题若不先收敛，会让系统集成状态看上去比实际更乐观。

**判断**

当前红灯不是“基础设施没搭起来”，而是“真实主链已经存在，且已经出现可复现回归”。这类问题应优先级高于继续补充外围模块。

**建议动作**

1. 先锁定 runtime/cognition 成功路径的最终响应合同，明确 response_text、status、observation projection、llm fallback 的优先级与拼装规则。
2. 以 [../../tests/integration/agent_loop/RuntimeUnaryIntegrationTest.cpp](../../tests/integration/agent_loop/RuntimeUnaryIntegrationTest.cpp) 和 [../../tests/integration/cognition/CognitionRuntimeIntegrationTest.cpp](../../tests/integration/cognition/CognitionRuntimeIntegrationTest.cpp) 为第一修复门，避免继续在主链上叠加新能力。
3. 在 runtime 详设中补一份“最终结果合同说明”，把 fixture path 与 true integration path 的预期差异写清。

### GINT-02（P0）knowledge 与 llm 已实现，但尚未成为默认 live unary 闭环的必需依赖

**问题描述**

knowledge 与 llm 子系统本身已经具备真实实现和 smoke 证据，但 runtime 当前的 live unary 准入条件并不要求 knowledge_service 或 llm_manager。这意味着系统现在拥有“可工作的 knowledge/llm 子系统”，却未拥有“默认必须经过 knowledge/llm 的主链”。

**证据**

1. [../../runtime/include/RuntimeDependencySet.h](../../runtime/include/RuntimeDependencySet.h) 定义了 knowledge_service 与 llm_manager，但 has_live_unary_ports 并不要求它们。
2. [../../knowledge/src/facade/KnowledgeService.cpp](../../knowledge/src/facade/KnowledgeService.cpp) 和 [../../llm/src/LLMManager.cpp](../../llm/src/LLMManager.cpp) 都已具有非占位实现。
3. [../../cognition/src/llm/CognitionLlmBridge.cpp](../../cognition/src/llm/CognitionLlmBridge.cpp) 证明 cognition 可以真实调用 llm。
4. knowledge 与 llm 相关 smoke 为绿，但 runtime 真主链仍可在缺失这些依赖时退化运行。

**影响**

1. “子系统实现已存在”并不自动等价于“系统默认闭环已经消费该子系统”。
2. 如果产品语义要求 knowledge/llm 属于默认能力，则当前 runtime 准入条件仍偏松。
3. 如果产品语义允许退化运行，则需要更明确地区分 mandatory path 与 degraded path，避免评审口径混淆。

**判断**

这不是一个简单的设计缺失，而是系统语义尚未完全收口：DASALL 需要明确 knowledge/llm 对默认单 Agent unary 路径到底是“可选增强”还是“默认必需”。

**建议动作**

1. 在 runtime 侧明确产品模式：默认必须带 knowledge/llm，或默认允许退化但需打清晰标记。
2. 若选择“默认必需”，则将 knowledge_service / llm_manager 纳入 has_live_unary_ports 或等价 gate。
3. 若选择“默认可退化”，则应补齐 degraded path 的对外语义、审计与测试断言。

### GINT-03（P1）knowledge 的结构化证据在 runtime/memory/contracts 交界面被扁平化

**问题描述**

knowledge 内部已经具备 EvidenceSlice、EvidenceBundle 与 context_projection 等更丰富的数据模型，但 runtime 和 memory 当前传递 external_evidence / retrieval_evidence 时仍以字符串数组为主。这导致 provenance、freshness、citation、version 等跨模块最有价值的信息在主链上丢失。

**证据**

1. [../../knowledge/include/KnowledgeTypes.h](../../knowledge/include/KnowledgeTypes.h) 定义了 EvidenceSlice、EvidenceBundle 与 context_projection。
2. [../../memory/include/context/MemoryContextRequest.h](../../memory/include/context/MemoryContextRequest.h) 仍以 external_evidence: vector<string> 作为输入。
3. [../../contracts/include/context/ContextPacket.h](../../contracts/include/context/ContextPacket.h) 中 retrieval_evidence 仍为文本数组投影。
4. [../../memory/src/context/ContextOrchestrator.cpp](../../memory/src/context/ContextOrchestrator.cpp) 也主要以字符串 evidence 组装 ContextPacket。
5. [../../tests/integration/knowledge/KnowledgeRetrievalSmokeTest.cpp](../../tests/integration/knowledge/KnowledgeRetrievalSmokeTest.cpp) 已证明 knowledge 内部可以保留 authority/citation 级语义。

**影响**

1. cognition 难以稳定做 freshness、trust、citation 级推理。
2. 后续若要补 explainability、引用回链、审计追踪，将更容易出现私有补丁字段扩散。
3. contracts 当前看似稳定，实则在最关键的 knowledge -> memory/cognition handoff 上过薄。

**判断**

本轮评审中，最值得进入 shared contracts 的新增面只有这一项：为 retrieval evidence 增加最小结构化投影，但保持 additive + optional，继续兼容现有文本路径。

**建议动作**

1. 在 contracts 层新增轻量 evidence projection，例如 evidence_ref、source_ref、source_kind、summary_text、trust_level、freshness、optional anchor_locator。
2. 不将 KnowledgeQuery、EvidenceSlice、EvidenceBundle 等内部 supporting objects 直接提升为 shared contracts。
3. 为 knowledge -> runtime -> memory/cognition 增加 citation preservation regression cases，确保结构化字段在主链上不丢失。

### GINT-04（P1）infra diagnostics 的 retained snapshot 回读路径出现真实回归

**问题描述**

infra diagnostics 的 execute -> store -> get_snapshot 设计与实现都已经存在，但 focused smoke 当前失败，说明 retained snapshot round-trip 不能被视作已闭合能力。

**证据**

1. [../../infra/src/diagnostics/DiagnosticsServiceFacade.cpp](../../infra/src/diagnostics/DiagnosticsServiceFacade.cpp) 已实现 execute、get_snapshot、export_snapshot 三条路径。
2. [../../tests/integration/infra/InfraDiagnosticsSmokeTest.cpp](../../tests/integration/infra/InfraDiagnosticsSmokeTest.cpp) 的 retained snapshot 回读断言当前失败。

**影响**

1. diagnostics 无法稳定作为 infra 侧保留型证据链使用。
2. 后续如果 runtime、access、ops 需要基于 retained snapshot 做诊断回放和问题定位，将缺失可靠基础。

**判断**

infra 的问题不是“基础模块未实现”，而是一个具体且高价值的 round-trip gate 已红。这类回归应在下一轮系统收敛中被单独处理，而不应被整体“infra 大体可用”的判断掩盖。

**建议动作**

1. 以 [../../tests/integration/infra/InfraDiagnosticsSmokeTest.cpp](../../tests/integration/infra/InfraDiagnosticsSmokeTest.cpp) 为第一修复门，查清 execute 后的 snapshot store、retention、get_snapshot 映射是否一致。
2. 将 diagnostics retained snapshot 明确纳入 infra 集成收口标准，避免只看 facade 是否存在。

### GINT-05（P2）tools -> services 的功能链已打通，但默认返回码语义不够一致

**问题描述**

tools 通过 ToolServiceBridge 与 capability services 真实连接，结果也可以投影为 Observation；但 BuiltinExecutorLane 的默认服务在返回成功 payload 时仍可能把 code 填成 ToolExecutionFailed，而 map_service_result 又主要依据 error 判断 success。这会造成跨层状态语义不统一。

**证据**

1. [../../tools/src/bridge/ToolServiceBridge.cpp](../../tools/src/bridge/ToolServiceBridge.cpp) 已把 ToolInvocationContext 映射为 execution/data/diagnose 请求。
2. [../../tools/src/execution/BuiltinExecutorLane.cpp](../../tools/src/execution/BuiltinExecutorLane.cpp) 中默认 execution/data service 的 success-like 返回仍复用了失败 code。
3. 同文件对 ToolResult.success 的判断主要依赖 error 是否为空，而非 contracts::ResultCode。
4. [../../tests/integration/tools/ToolServicesSmokeIntegrationTest.cpp](../../tests/integration/tools/ToolServicesSmokeIntegrationTest.cpp) 与 capability services smoke 为绿，说明这不是链路断裂，而是语义一致性问题。

**影响**

1. 短期不会阻断 tools -> services 链路，但长期会污染统一错误处理、审计和指标解释。
2. 一旦更多消费者开始读取 ResultCode，当前“看 error 不看 code”的习惯会变成隐藏故障源。

**建议动作**

1. 清理默认 service 返回码，让 code 与 error 语义一致。
2. 在 tools/capability services 的 contract 测试中同时断言 success、error、code 三元一致性。

### GINT-06（P2）部分设计文档的“现状判断”已落后于仓库事实

**问题描述**

本轮核查确认，部分旧评审与子系统详设中的“当前缺失项”已经被后续代码实现追上。例如 llm 公共头与 manager 已经存在，runtime 主链也已从 placeholder 走向真实 orchestrator。这意味着部分旧结论需要从“未实现”修正为“已实现但存在集成回归”。

**影响**

1. 若继续按旧文档判断优先级，容易误把“真实回归”当作“尚未开工”。
2. 这会直接影响任务拆解顺序和收敛策略。

**建议动作**

1. 把带“现状/缺口”性质的章节纳入 doc-to-repo 一致性校验。
2. 评审文档中对实现状态的结论应标注 evidence date，避免长期漂移。

## 5. 已确认的正向集成证据

### 5.1 入口链并非纸面设计

access、cli、daemon 三者之间的本地控制面入口链已经具备真实实现与通过的 smoke 证据：

1. [../../apps/cli/src/main.cpp](../../apps/cli/src/main.cpp) 保持纯客户端角色，通过本地 IPC/UDS 访问 daemon。
2. [../../apps/daemon/src/main.cpp](../../apps/daemon/src/main.cpp) 负责将 access 请求投影为 runtime AgentRequest，并映射回对外 dispatch 结果。
3. [../../access/src/RuntimeBridge.cpp](../../access/src/RuntimeBridge.cpp) 对 normalizer 和 decision proof 做 fail-closed 校验，再转入 runtime。
4. 代表性测试 DaemonUnaryRuntimeBridgeTest、CliDaemonSocketPathIntegrationTest、DaemonBinaryUnarySmokeTest 为绿。

### 5.2 tools -> capability services 是真实链路，不是未来规划

1. [../../tools/src/bridge/ToolServiceBridge.cpp](../../tools/src/bridge/ToolServiceBridge.cpp) 已真实构造 execution/data/diagnose 请求。
2. [../../services/src/ServiceFacade.cpp](../../services/src/ServiceFacade.cpp) 已作为 capability services 的统一门面存在。
3. ToolServicesSmokeIntegrationTest 与 CapabilityServicesSmokeIntegrationTest 为绿，证明 tool execution 不依赖“文档假设”即可贯穿到 capability services。

### 5.3 platform_linux、knowledge、llm 的独立 smoke 已能提供正向信号

1. LinuxPlatformBootstrapIntegrationTest 为绿，说明 platform_linux 作为本地控制面与启动基座是稳定的。
2. dasall_knowledge_retrieval_smoke_integration_test 为绿，说明 knowledge 的路由、召回、重排、证据组装主流程已具备最小可信度。
3. LLMSubsystemSmokeIntegrationTest 为绿，说明 llm 自身不是空壳；问题主要在系统默认主链如何消费它。

### 6.2 分系统补充判断

#### access

1. 公共数据结构已经具备 sidecar 所需的主体身份、授权证明与 handoff request，不再只是概念对象。
2. RuntimeBridge 对 normalizer 和 decision proof 做 fail-closed 判断，符合 access 作为入口防线的职责。
3. 当前 access 的主要风险不在链路未接通，而在后续要不要继续扩展更多协议面；就当前 unary 主路径而言，已经足够稳定。

#### capability services

1. services 的数据结构与门面已经能被 tools 真正消费，这一点比“文档里有接口设计”更重要。
2. 当前问题主要来自 tools 侧对返回码的一致性处理，而非 services façade 缺位。

#### cli

1. cli 保持纯客户端，不直接侵入 runtime 或 memory，设计与实现边界一致。
2. 当前可以把 cli 视为一个较低风险的消费层，而不是系统主链阻塞点。

#### cognition

1. cognition 自身的门面、LLM bridge、response builder 都不是空实现。
2. 当前的主要风险集中在与 runtime 汇合后的最终 response contract，而不是 cognition 子系统本身毫无落地。

#### contracts

1. contracts 层总体保持了克制，没有出现大规模把 module-local supporting objects 推进 shared layer 的问题。
2. 真正需要新增的 shared 面很少，最重要的是 retrieval evidence 的轻量结构化投影，而不是扩张更多 planning、tool、access supporting objects。

#### daemon

1. daemon 已成为实际的本地控制面装配根，而不是计划中的未来组件。
2. 当前结论可以明确为：daemon 已经具备与 access/runtime 共同支撑 unary 请求的能力。

#### infra

1. infra 不能再被笼统归类为“尚未进入集成阶段”，因为 diagnostics façade 和 smoke 都已存在。
2. 但只要 retained snapshot round-trip 仍为红灯，就不能简单宣布 diagnostics ready。

#### knowledge

1. knowledge 内部的数据模型和检索主流程已经较丰富，且 smoke 已给出正向证据。
2. 其核心短板不在子系统内部，而在主链出口：当前缺少一条更薄但更可靠的 shared evidence projection。

#### llm

1. llm 当前是“实现存在、系统消费策略未完全收口”的典型。
2. 若未来产品语义要求默认必须经过 llm，则 runtime 的 dependency gate 与 degraded path 说明必须同步升级。

#### memory

1. memory 已被 runtime 主链真实调用，说明 ADR-006 的上下文装配边界已经进入代码面。
2. 当前不足主要体现在 external_evidence / retrieval_evidence 的语义过薄，而非 memory façade 缺位。

#### runtime

1. runtime 是本轮评审最关键的子系统，因为它既证明了“系统已从骨架演进到真实主控”，也暴露了“主链仍未达到集成就绪”。
2. 当前任何“主链 ready”结论都必须以 RuntimeUnaryIntegrationTest 与 CognitionRuntimeIntegrationTest 重新转绿为前提。

#### tools

1. tools 已经是真正的执行治理中心，而不是单纯的 adapter 集合。
2. 当前优先级更高的是清理结果码语义与观测语义，而不是重构整体架构。

#### platform_linux

1. platform_linux 当前提供的是稳定的本地控制面与启动基础，不是本轮系统风险中心。
2. 它的价值在于为 cli、daemon、access 提供一个已验证的基础平台层。

## 7. 其它有价值的评估维度

### 7.1 配置治理

1. RuntimePolicySnapshot 的总体方向仍然正确，profiles 作为编译与运行配置治理中枢的定位没有偏离。
2. 当前缺口不在“有没有 snapshot”，而在“系统级消费矩阵是否足够清晰”，尤其是多模块共享键的 semantic owner、primary consumer、override 行为与 hot-reload 语义。
3. 若没有这份系统级消费矩阵，后续实现很容易把同一配置键在多个模块中解释成不同含义。

### 7.2 恢复治理

1. ADR-007 的大方向已守住：cognition 提供建议，runtime 持有最终恢复执行权。
2. 当前仍需补强的是 runtime 侧恢复上下文的显式边界表，特别是 retry budget、idempotency、side-effect classification、circuit state、deadline 等执行控制信息的归属。
3. 若这一层继续依赖“实现时心照不宣”，后续 failure injection 与恢复拒绝原因将难以统一验证。

### 7.3 可观测性与审计

1. access、tools、infra、runtime、llm 都已体现出较强的 observability 意识，说明系统并非只追求 happy path。
2. 但系统级的关键事件矩阵仍需进一步清晰，例如 access deny、runtime recovery reject、knowledge degrade、tool compensation hint、diagnostics snapshot retention 等事件最好具备统一命名与保留规则。

### 7.4 测试治理

1. 本轮评审最明显的积极信号是：系统中已经存在一批真正有价值的 focused integration tests，而不是只有单测或空 smoke。
2. 当前问题不是“没有测试”，而是“关键 gate 已经开始揭示真实回归”。
3. 因此下一阶段应优先修复红灯 gate，而不是以更多新测试名称掩盖现有失败。

### 7.5 文档治理

1. 旧版评审对系统级边界判断仍有价值，但对实现状态的部分结论已落后于仓库事实。
2. 从 Design -> Build 双轨治理看，文档事实漂移本身就是工程风险，应被纳入持续校验，而不只是评审时补记一条备注。

## 9. 风险排序与收敛建议

### 9.1 P0：必须先收敛的事项

1. 修复 runtime/cognition 最终响应合同，使 RuntimeUnaryIntegrationTest 与 CognitionRuntimeIntegrationTest 重新转绿。
2. 明确 knowledge/llm 在默认 live unary 路径中的产品语义，统一 dependency gate 与 degraded path 说明。
3. 把 runtime unary true integration gate 正式提升为系统级放行条件，而不是仅作为子系统内建议测试。

### 9.2 P1：下一阶段必须补强的事项

1. 为 retrieval evidence 增加最小结构化投影，避免 knowledge 证据在 contracts、memory、runtime 边界上被过度扁平化。
2. 修复 infra diagnostics retained snapshot 回读回归，并将其作为 infra 集成收口项。
3. 将 RuntimePolicySnapshot 消费矩阵与 Recovery Context 边界表提升为系统级 SSOT，而不是仅散落在子系统文档中。

### 9.3 P2：质量治理类持续工作

1. 清理 tools/capability services 的 ResultCode、error 语义不一致问题。
2. 为带“现状与缺口”性质的设计文档增加 doc-to-repo 一致性校验。
3. 梳理系统级关键事件与指标命名矩阵，减少后续 observability 漂移。

## 10. 放行判断

### 10.1 可以放行继续推进的部分

1. access、cli、daemon、platform_linux、capability services 的当前设计与实现可以继续按既有方向推进。
2. tools、knowledge、llm、memory 可以继续做子系统内增强，但前提是避免改变当前已验证的 public surface 或破坏主链契约。
3. contracts 可以继续保持克制冻结策略，优先只推进最小结构化 evidence projection。

### 10.2 当前不应宣布 ready 的部分

1. runtime 默认单 Agent unary 主链，不应在 RuntimeUnaryIntegrationTest 与 CognitionRuntimeIntegrationTest 失败时被宣布为 integration ready。
2. infra diagnostics retained snapshot 能力，不应在 InfraDiagnosticsSmokeTest 失败时被视作稳定基线。
3. “knowledge/llm 已完整纳入默认主链”这一判断当前并不成立。

### 10.3 现阶段最合理的阶段性结论

1. DASALL 已经越过“纯详细设计阶段”，因为多个入口链、能力链、平台链与子系统 smoke 都给出了真实实现证据。
2. DASALL 尚未越过“系统主链集成收口阶段”，因为 runtime unary 的 true integration gate 和 infra diagnostics retained snapshot gate 仍为红灯。
3. 因此，当前阶段性结论应明确表述为：架构可行、若干集成切片可运行、系统主链待收口。

---

## 11. 最终结论

1. DASALL 当前最值得肯定的，不是某一个子系统“写得很完整”，而是 access、daemon、tools、knowledge、llm、platform_linux 等关键切片已经能给出真实代码与测试证据，证明这不是停留在纸面的架构工程。
2. DASALL 当前最需要警惕的，也不是“子系统都还没做”，而是 runtime 主链已经进入真实集成阶段，并且已经出现清晰、可复现、值得优先修复的红灯 gate。
3. 如果从集成角度而不是文档角度做决策，本阶段的核心任务已经非常明确：先把单 Agent unary 主链真正打绿，再补结构化证据投影与 diagnostics retained snapshot，最后再扩更多高级能力。
4. 换言之，DASALL 现在的正确节奏不是继续扩面，而是先收口。只有把这几个系统级红灯关掉，后续更大规模的并行开发才有稳定基座。
