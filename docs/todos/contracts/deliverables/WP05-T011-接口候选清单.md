# WP05-T011 跨模块接口候选清单

最近更新时间：2026-03-21

上游输入：DASALL contracts 冻结实施计划阶段 5、DASSALL Agent 架构文档 5.4/7.3/7.4、DASALL Engineering Blueprint 接口分布建议、ADR-006、ADR-008

## 0. Phase 0 研究学习证据链

### 本地证据清单

1. `docs/plans/DASALL_contracts冻结实施计划.md` 阶段 5 已冻结接口准入原则：只有真正跨模块且需要作为稳定依赖面的抽象接口，才进入 contracts；不得为了“看起来解耦”而把所有接口都塞进 contracts。
2. `docs/architecture/DASSALL_Agent_architecture.md` 第 7.4 节已冻结模块依赖规则：runtime 依赖 llm、tools、memory、services 抽象；cognition 只能依赖 contracts 和抽象接口，说明接口候选必须围绕跨模块依赖倒置面识别。
3. `docs/architecture/DASSALL_Agent_architecture.md` 第 5.4.1 节与第 7.3 节给出 `ILLMAdapter`、`IPlanner`、`IToolManager`、`IMemoryStore` 等接口草图，说明仓库已经存在可追溯的接口命名与输入输出约束来源。
4. `docs/architecture/DASALL_Engineering_Blueprint.md` 已给出建议接口文件分布：`IToolManager.h`、`IMemoryStore.h`、`IContextOrchestrator.h`、`IKnowledgeService.h`、`IExecutionService.h`、`IDataService.h`、`IAgentRegistry.h`、`IResultMerger.h`，可作为候选目录的直接命名锚点。
5. ADR-006 已冻结 ContextOrchestrator 只负责语义上下文编排、PromptComposer 只负责消息装配，说明 `IContextOrchestrator` 属于跨 memory/runtime/llm 的稳定治理面，而 `PromptComposer` 仍更接近 llm 子系统内部实现面。
6. ADR-008 已冻结 AgentOrchestrator 与 MultiAgentCoordinator 的控制权分层，说明 runtime 与 multi_agent 之间至少存在 `IAgentRegistry`、`IResultMerger` 这类跨模块协同接口候选，但它们仍受上游协同对象冻结程度约束。

### 外部参考清单

1. Microsoft Azure Architecture Center, Strangler Fig pattern：增量迁移应优先维持稳定门面与渐进替换路径，避免让调用方感知内部迁移细节；可直接映射为“只有稳定依赖面才值得抽入共享 contracts”。
2. Protocol Buffers Best Practices：客户端和服务端从不保证同步升级，因此共享契约必须优先兼容演进，避免把高频变化的内部细节暴露为长期共享接口。

### 对本任务的可落地启发

1. 本轮不应实现真实接口体，只应冻结“候选目录”和最小元数据，因为阶段 5 的目标是先识别稳定依赖面，再由后续 T012 做准入评估。
2. 候选目录必须剔除平台、基础设施、协议适配、工具子系统内部实现接口，避免把局部实现替换点误当成跨模块稳定依赖面。
3. 候选目录需要显式记录 owner、primary consumer、readiness 与 stable anchor，才能支撑 T012 的程序化准入守卫。
4. 当前最接近 admission review 的候选主要集中在已拥有冻结输入输出契约的 llm/tools 子域，其余接口虽然跨模块，但仍需要等待 supporting contracts 完整冻结。

## 1. 任务理解

本任务只处理 WP05-T011：

1. 识别进入 contracts 的跨模块接口候选。
2. 只保留稳定依赖面，剔除模块内部或协议/平台细节接口。
3. 新增 `contracts/include/boundary/InterfaceCatalog.h`，把候选目录程序化。
4. 新增并注册 `tests/contract/smoke/InterfaceCatalogContractTest.cpp`。

本任务不处理：

1. 真实接口方法签名的最终冻结与头文件落盘。
2. `InterfaceAdmissionGuards.h` 的准入结论规则实现。
3. platform/infra/tools 内部适配接口的 contracts 化。

## 2. 约束与边界

### 2.1 应保留的候选面

1. 跨顶层模块边界存在明确 owner 与 consumer。
2. 输入输出或语义锚点可追溯到架构文档、ADR 或 Blueprint。
3. 接口目的是隔离模块实现细节，而不是暴露协议/平台细节。
4. 可以作为后续 contracts admission review 的独立条目审查。

### 2.2 应剔除的接口类型

1. 平台抽象接口，如 `IGPIO`、`IUART`、`INetwork`。
2. 基础设施观测接口，如 `ILogger`、`ITracer`、`IMetricsExporter`。
3. 协议与工具子系统内部接口，如 `IMCPAdapter`、`IPolicyGate`、`ICapabilityCache`。
4. 只服务单模块内部装配、未形成稳定外部依赖面的接口。

## 3. Design 原子清单

1. D1：锁定 T011 的接口候选筛选规则。
- 输入依据：阶段 5 接口设计原则、架构 7.4 模块依赖规则、Blueprint 接口分布建议。
- 产出路径：本文件第 2 节与第 4 节。
- 完成判定：候选目录只保留跨模块稳定依赖面，并显式给出剔除边界。
- 风险与回退：若混入 platform/infra/internal adapter 接口，则回退到“owner + consumer + stable anchor”三条件过滤。

2. D2：形成可程序化审查的候选目录字段模型。
- 输入依据：T011-B 代码目标、T012-B 准入守卫前置需要。
- 产出路径：`contracts/include/boundary/InterfaceCatalog.h`。
- 完成判定：目录至少包含接口名、owner、primary consumer、readiness、stable anchor、rationale。
- 风险与回退：若目录只剩名称列表而无审查元数据，则无法支撑 T012，需回退补齐字段模型。

3. D3：锁定首版候选集与 review-ready 范围。
- 输入依据：架构文档接口草图、Blueprint 文件分布建议、已冻结 contracts 对象。
- 产出路径：本文件第 4 节和 `InterfaceCatalog.h` 的静态表。
- 完成判定：候选集完整覆盖 llm/tools/memory/knowledge/services/multi_agent/cognition 的跨模块稳定依赖面，并区分 `ReviewReady` 与 `AwaitingSupportingContracts`。
- 风险与回退：若把尚无跨模块 consumer 的模块内部接口纳入，则回退删除。

## 4. 首版候选目录

| 候选接口 | owner 模块 | primary consumer | readiness | stable anchor | 保留理由 |
|---|---|---|---|---|---|
| `IPlanner` | cognition | runtime | AwaitingSupportingContracts | `GoalContract`、`ContextPacket`、`Observation` | runtime 需要通过抽象规划面消费 cognition，而不是依赖具体 Planner 实现；但 `PlanGraph`/`ReplanResult` 尚未冻结 |
| `IToolManager` | tools | runtime | ReviewReady | `ToolRequest`、`ToolResult`、`ToolDescriptor` | 工具治理已形成稳定请求/结果/目录对象，具备进入 admission review 的最小条件 |
| `ILLMAdapter` | llm | runtime | ReviewReady | `LLMRequest`、`LLMResponse` | llm 统一调用面已具备冻结请求/响应对象与架构草图 |
| `IMemoryStore` | memory | runtime | AwaitingSupportingContracts | `Turn`、`Session`、`SummaryMemory`、`MemoryFact` | memory 对象已部分冻结，但 `SessionSnapshot`/`RetrievalQuery` 等 supporting objects 尚未统一 |
| `IContextOrchestrator` | memory | runtime | AwaitingSupportingContracts | `ContextPacket`、ADR-006 责任链 | 该接口是 memory/runtime/llm 的稳定治理面，但请求/结果对象尚未单独冻结 |
| `IKnowledgeService` | knowledge | runtime | AwaitingSupportingContracts | `RetrievalResult` 语义锚点 | knowledge 对外检索面明确存在，但请求/结果对象仍未在 contracts 中冻结 |
| `IExecutionService` | services | tools | AwaitingSupportingContracts | “Execution Service 对上暴露统一执行动作语义” | tools 到 services 的核心隔离面，适合作为共享 contracts 候选 |
| `IDataService` | services | tools | AwaitingSupportingContracts | Blueprint `IDataService.h` | 数据能力对 tools/runtime 的统一入口明确存在，但 supporting objects 尚未冻结 |
| `IAgentRegistry` | multi_agent | runtime | AwaitingSupportingContracts | ADR-008 协同责任链、Worker 能力匹配语义 | runtime 依赖 registry 进行 worker 能力匹配，是稳定协同依赖面 |
| `IResultMerger` | multi_agent | runtime | AwaitingSupportingContracts | ADR-008 结果汇聚语义 | runtime 需要消费结构化协同汇聚能力，但结果对象族仍待继续冻结 |
