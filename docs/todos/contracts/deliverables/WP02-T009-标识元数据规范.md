# WP02-T009 标识元数据规范

最近更新时间：2026-03-14
任务状态：In Review
任务编号：WP02-T009
上游输入：计划文档第 8 阶段 1、架构文档

## 1. 任务理解

本任务只处理 WP02-T009：统一 request_id、session_id、trace_id、task_id、lease_id 规则，产出标识元数据规范。

本任务完成判定要求：明确命名、唯一性、传播范围和父子关系。

本任务不扩展到时间字段（T010）、事件封套（T011）或代码实现。

## 2. 约束与边界

### 2.1 可追溯约束

1. 来自计划文档阶段 1：通用标识与追踪元数据应统一约定，至少涉及 request_id、session_id、trace_id、task_id、parent_task_id、lease_id。
2. 来自架构文档：入口阶段需要生成 request_id/session_id/trace_id；多 Agent 日志至少包含 trace_id、task_id、lease_id、parent_task_id。
3. 来自 ADR-008：WorkerTask 必须明确 task_id、parent_task_id、lease_id，且子任务标识不能越权成为顶层主控标识。
4. 来自 contracts 冻结原则：语义优先于 schema，兼容优先，禁止跨目录重复发明同类标识字段语义。

### 2.2 边界

1. 本任务只定义标识元数据语义规则，不定义传输协议实现。
2. 本任务只规范五个主标识与 parent_task_id 关系，不定义业务对象私有 ID。
3. 不改写 ADR-006/007/008 的职责结论。

### 2.3 非目标

1. 不定义 created_at/deadline_at/timeout_ms/ttl（属于 T010）。
2. 不定义 EventEnvelope 字段（属于 T011）。
3. 不编写 contracts/runtime 代码。

## 3. 方案对比与决策

### 3.1 方案 A：扁平自由命名

设计思路：各模块按需要自由命名和生成标识，只要求字段名近似一致。

优点：

1. 改动少，短期接入快。

缺点：

1. 难保证全链路追踪一致性。
2. parent_task_id 与 lease_id 语义易漂移。
3. 无法稳定满足唯一性和传播范围审计。

### 3.2 方案 B：分层标识模型 + 显式传播规则

设计思路：定义五类标识的命名模板、唯一性域、传播范围和父子关系约束，统一日志与契约消费口径。

优点：

1. 直接满足 T009 完成判定。
2. 与阶段 1 目标和多 Agent 追踪要求一致。
3. 能被 T010/T011/T013 直接消费。

缺点：

1. 需要评审并统一生成器策略。

### 3.3 决策

采用方案 B。

取舍理由：在不进入实现细节的前提下，方案 B 最小化语义歧义并最大化可追溯性。

## 4. 最终产出

### 4.1 标识定义总表

| 标识 | 命名规则 | 唯一性域 | 传播范围 | 父子关系 |
|---|---|---|---|---|
| request_id | 请求级唯一 ID，建议前缀 req_ | 全局请求级唯一 | 从入口到最终结果全链路透传 | 顶层根标识，不挂父级 |
| session_id | 会话级唯一 ID，建议前缀 sess_ | 会话空间唯一 | 同一会话所有 turn/任务透传 | request_id 归属到某 session_id |
| trace_id | 分布式追踪 ID，建议前缀 trc_ | 追踪链路唯一 | 日志、事件、工具与多 Agent 轨迹全链路透传 | request_id 与 task_id 共享同一 trace_id |
| task_id | 任务单元 ID，建议前缀 task_ | 任务图内唯一，跨会话可全局唯一 | Runtime 任务、WorkerTask、审计日志透传 | 可有 parent_task_id，形成任务树/DAG |
| lease_id | 租约实例 ID，建议前缀 lease_ | task_id 下唯一，建议全局唯一 | 仅在调度、执行、回收相关链路透传 | 归属于 task_id，不作为父任务标识 |

补充关系字段：

| 字段 | 语义 | 约束 |
|---|---|---|
| parent_task_id | 当前 task 的父任务标识 | 仅用于 task 级层次关系；顶层 task 为空 |

### 4.2 传播与继承规则

1. request_id、session_id、trace_id 在入口生成后不得在子链路重写。
2. task_id 在任务创建时生成；子任务必须设置 parent_task_id=父 task_id。
3. lease_id 在调度租约授予时生成；同一 task 的不同执行尝试可有不同 lease_id。
4. Worker 输出回传时必须携带 task_id、parent_task_id、lease_id、trace_id，便于回放。
5. 任一日志或事件若包含 task_id，必须同时可关联 trace_id；否则视为不可追踪数据。

### 4.3 唯一性与校验规则

1. request_id：单请求唯一，不可复用到其他请求。
2. session_id：会话生命周期内稳定，不因重试切换。
3. trace_id：一次端到端执行链路稳定，跨子任务不变。
4. task_id：同一 TaskGraph 内不可重复；若重规划生成新 task，必须新建 task_id 并记录 parent_task_id。
5. lease_id：同一调度租约周期唯一；租约续期应保持 lease_id 或显式生成新 lease_id 并记录继承关系。

### 4.4 命名与格式建议

1. 字段名统一 snake_case：request_id/session_id/trace_id/task_id/parent_task_id/lease_id。
2. 值格式建议使用小写字母、数字与下划线，避免空格与大小写混用。
3. 不在 ID 字符串中编码业务语义，防止后续演进时被解析依赖绑定。

### 4.5 与后续任务映射

1. T010 可基于本规范补充时间传播规则（如 created_at 与 timeout_ms 的关联维度）。
2. T011 可复用 trace_id/request_id 作为 EventEnvelope 头部候选字段。
3. T013 可把“唯一性校验+传播完整性”固化为 review checklist。

## 6. 风险与回退

1. 风险：模块侧继续私有生成 trace_id，导致链路断裂。
   回退：入口统一生成并强制透传，子模块只消费不重写。
2. 风险：parent_task_id 在并行拆分中遗漏，造成任务树不可回放。
   回退：任务创建流程增加必填校验；缺失时阻断派发。
3. 风险：lease_id 与 task_id 绑定关系被错误复用。
   回退：租约层单独生成 lease_id，并记录 task_id 归属。
