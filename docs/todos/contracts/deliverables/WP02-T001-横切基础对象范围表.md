# WP02-T001 横切基础对象范围表

最近更新时间：2026-03-14
任务状态：In Review
任务编号：WP02-T001
上游输入：WP01-T013 M1 冻结包

## 1. 任务理解

本任务只回答一个问题：WP-02 在进入字段设计、规则设计和对象头文件落盘之前，哪些语义类别被视为“横切基础对象”并纳入统一冻结范围。

本任务不定义字段全集，不定义错误码分类，不定义具体枚举值，不定义目录内最终头文件拆分方式，只冻结“范围边界”和“后续任务可消费的对象清单”。

本任务完成后，WP-02 后续任务必须在本范围内细化，不得绕开本范围自行新增新的基础语义类别。

## 2. 约束与边界

### 2.1 直接约束

1. 来自实施计划阶段 1：横切基础对象冻结的优先对象明确包含 error、event、RuntimeBudget、通用标识与追踪元数据、通用时间与超时表达。
2. 来自 WP-02 TODO：本任务完成判定必须明确纳入 error、event、budget、id、time。
3. 来自 WP01-T013 M1 冻结包：后续任务不得回退主对象边界结论，只能在已冻结边界上继续细化。
4. 来自工程蓝图：contracts 顶层目录已显式包含 event/、error/、checkpoint/，其中 checkpoint/ 承载 RuntimeBudget；Request Normalizer 与多 Agent 链路已显式依赖 request_id、session_id、trace_id、task_id、lease_id；主循环防护项和运行治理要求显式依赖 timeout/deadline 类语义。
5. 来自 ADR-006/007/008：runtime_budget、token_budget_report、Checkpoint、RuntimeBudget、parent_request_id、parent_task_id、lease_id、timeout 等语义已被冻结边界引用，但 ADR 未要求在本任务中展开字段设计。

### 2.2 本任务边界

1. 只定义语义类别范围，不冻结字段名和字段类型。
2. 只定义“应统一收口”的横切对象，不下钻到子域专属 payload。
3. 不改写 ADR-006/007/008 的职责分层结论。
4. 不提前进入 T002 兼容性规则、T004 错误码分类、T007 Budget 字段、T009 标识规则、T010 时间规则、T011 EventEnvelope 头部字段设计。

### 2.3 非目标

1. 不定义 ErrorInfo 必填字段全集。
2. 不定义 ResultCode 分类树。
3. 不定义 RuntimeBudget 与 BudgetSnapshot 的字段细节。
4. 不定义 EventEnvelope 头部字段集合。
5. 不定义 ID 生成算法、编码格式或序列化方案。
6. 不定义 created_at、deadline_at、timeout_ms、ttl 的最终字段命名。

## 3. 方案对比与决策

### 3.1 方案 A：最小闭包范围

范围仅纳入五类横切基础语义：

1. error
2. event
3. budget
4. id
5. time

优点：

1. 与 WP02-T001 完成判定完全一致，可追溯性最强。
2. 不提前侵入字段设计和实现细节，符合“一个任务只交付一个明确产出”。
3. 能为 T004、T007、T009、T010、T011 提供稳定上位边界，且不制造额外评审面。

缺点：

1. trace、deadline、timeout、lease 等具体对象暂时只能作为 id/time/budget 的子类语义，不能在本任务中独立成章。
2. 后续任务需要继续把五类语义拆成对象级规则，文档粒度偏粗。

### 3.2 方案 B：扩展闭包范围

在方案 A 基础上额外把 trace、audit、cancel/deadline、metadata/envelope header 作为独立横切类别冻结。

优点：

1. 后续字段设计入口更细，工程感更强。
2. 对日志、审计、取消链路的显式性更高。

缺点：

1. 超出 WP02-T001 完成判定，属于跨任务扩张范围。
2. trace、audit、cancel/deadline、metadata 在当前输入中仍依附于 id/time/event/budget 语义，没有被要求单独冻结为一级类别。
3. 会提前吞并 T009、T010、T011 的部分工作，破坏 TODO 原子任务边界。

### 3.3 决策

选择方案 A。

决策理由：

1. 方案 A 与计划文档阶段 1、总 TODO 的 M2 Gate、WP-02 T001 完成判定完全对齐。
2. 方案 A 满足“不允许跨任务扩张范围”的硬约束。
3. 方案 A 能把后续对象设计锁定在统一上位语义内，同时保留后续任务的评审颗粒度。

### 4.2 范围边界说明

1. budget 归入横切基础对象，但其物理归档位置可以继续落在 checkpoint/ 或独立预算文件中；本任务只冻结其横切语义地位，不冻结最终目录布局。
2. id 包含追踪与父子关联标识，但不把 TraceContext 整体扩张为独立一级类别；其具体结构留给 T009 和后续对象头文件设计。
3. time 包含 created_at、timestamp、deadline、timeout、ttl 等表达，但当前只冻结其必须统一收口，不冻结哪个对象必须携带哪些时间字段。
4. event 只冻结“通用封套”语义，不允许在本任务中引入模块私有 payload 细节。
5. error 只冻结统一失败语义，不把恢复策略本身纳入本任务；恢复裁定仍由 ADR-007 保持在 runtime/cognition 分层之下。

## 6. 风险与回退

### 6.1 主要风险

1. 风险：后续任务把 trace、audit、cancel、metadata 误当成新的一级横切类别。
   应对：统一要求这些语义先归并到 id、time、event、budget 的子类规则中，如需升级必须在 T013/T014 评审中显式决议。
2. 风险：budget 的物理目录归属被误读为只属于 checkpoint，导致其他对象重新发明局部预算字段。
   应对：在 T007 明确“RuntimeBudget 是横切语义，checkpoint 只是当前承载位置”。
3. 风险：事件头部设计提前混入模块私有 payload，削弱 event 封套的通用性。
   应对：在 T011 评审时严格使用“封套/载荷分层”检查项。

### 6.2 回退策略

1. 若后续评审发现五类范围不足以覆盖已冻结架构必需语义，不直接在 T001 内扩类，先登记为阻塞项并回到 WP-02 评审决议。
2. 若后续任务发现某语义类别只是某子域私有需求，应回退到对应子域任务处理，不得继续占用横切基础对象名额。
3. 若出现与 ADR-006/007/008 的冲突，以 ADR 冻结结论为准，撤回本范围表中任何越界解释，但不改动五类顶层纳入结论。
