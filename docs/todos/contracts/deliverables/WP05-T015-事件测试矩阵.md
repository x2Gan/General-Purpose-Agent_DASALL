## 0. Phase 0 研究学习证据链

### 本地证据清单

1. `contracts/include/event/EventEnvelope.h` 已冻结 EventEnvelope 结构：header 只承载公共元数据，payload 承载模块私有语义。
2. `contracts/include/event/EventEnvelopeGuards.h` 已冻结校验规则：header 六字段必填、`event_version > 0`、`header_keys` 必须命中白名单、payload_type/payload_json 必填。
3. `tests/contract/event/EventEnvelopeContractTest.cpp` 仅覆盖基础正例与私有 header key 负例，尚未形成面向“兼容演进”的系统矩阵。
4. `docs/todos/contracts/WP-05-子域细化与ContractTestsTODO.md` 对 T015-B 的目标路径已固定为 `tests/contract/event/EventEnvelopeCompatibilityContractTest.cpp`，范围应保持 event compatibility，不扩张到其他子域。
5. WP05 完成标准要求覆盖事件封套兼容性，因此 T015 需要显式证明“头部稳定 + payload 扩展约束”。

### 外部参考清单

1. Protobuf Language Guide（https://protobuf.dev/programming-guides/proto3/#updating）：向后兼容演进应允许旧端忽略未知新增字段，可映射为“封套解析忽略未知非头部字段”。
2. CloudEvents Spec（https://github.com/cloudevents/spec/blob/main/cloudevents/spec.md）：事件上下文有 REQUIRED 与 OPTIONAL/extension 元数据，消费者对 OPTIONAL 特性推荐忽略，可映射为“payload 扩展可放行，但核心头部必须稳定且可校验”。

### 对本任务的可落地启发

1. 兼容测试应区分两类扩展：非头部未知字段（可忽略）与 header 新键（应拒绝）。
2. payload 扩展应通过 payload_json 承载，不能把扩展字段提升到 header。
3. 负例必须覆盖 required 缺失、event_version 非法、header 白名单越权。
4. 断言应复用 `validate_event_envelope`，避免测试侧二次实现规则。

## 2. Design 原子清单

1. D1：定义 EventEnvelope 核心头部稳定矩阵。
- 输入依据：`EventEnvelope.h`、`EventEnvelopeGuards.h`。
- 产出路径：本文件第 3 节矩阵 + compatibility test。
- 完成判定：至少覆盖 required header、version 边界、header 白名单断言。
- 风险与回退：若只保留 happy-path，补齐 required/version/header-key 负例。

2. D2：定义 payload 扩展兼容矩阵。
- 输入依据：WP05-T015 目标（payload 扩展约束）。
- 产出路径：compatibility test。
- 完成判定：至少覆盖“未知非头部字段可忽略”与“payload_json 扩展可通过”。
- 风险与回退：若扩展场景被误放进 header，则回退为 payload-only 扩展路径。

3. D3：锁定 Build 三件套与发现性证据。
- 输入依据：`tests/contract/CMakeLists.txt` 注册路径。
- 产出路径：`tests/contract/CMakeLists.txt`。
- 完成判定：`ctest --test-dir build-ci -N -R EventEnvelopeCompatibilityContractTest` 构建后可发现 1 个测试。
- 风险与回退：若不可发现，先修注册，再检查实现。
