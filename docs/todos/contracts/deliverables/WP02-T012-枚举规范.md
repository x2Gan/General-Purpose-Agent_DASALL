# WP02-T012 枚举规范

最近更新时间：2026-03-14
任务状态：In Review
任务编号：WP02-T012
上游输入：WP02-T002-兼容性规则-v1.md、WP02-T003-字段演进规则表.md

## 1. 任务理解

本任务只处理 WP02-T012：定义枚举默认值与弃用规则，形成可评审的枚举规范。

本任务完成判定要求：每个枚举保留 Unspecified 值，并定义 deprecate 方式。

本任务不扩展到具体业务枚举全集落盘，不进入代码实现。

## 2. 约束与边界

### 2.1 可追溯约束

1. 来自 T002：新增优于修改、废弃优于删除、语义变化视为 breaking。
2. 来自 T003：枚举默认值必须保留未指定值；枚举值语义重解释属于 breaking。
3. 来自计划文档：所有枚举必须预留未指定值；废弃只能走 deprecate 与版本迁移。
4. 来自 WP-02 TODO：T012 必须明确 Unspecified 默认值与 deprecate 方式。

### 2.2 边界

1. 本任务只定义“枚举层通用规则”，不定义任一模块私有枚举清单。
2. 本任务不定义版本号升位细则，只定义枚举演进口径。
3. 不改写 ADR-006/007/008 边界结论。

### 2.3 非目标

1. 不定义错误码编码段（已在 T004 讨论）。
2. 不定义字段级完整 schema（由后续对象任务承接）。
3. 不编写 contracts/event/error 等代码。

## 3. 方案对比与决策

### 3.1 方案 A：弱约束枚举规则

设计思路：仅要求“尽量保留默认值、尽量不删值”，不设生命周期与评审门禁。

优点：

1. 规则少，落地阻力低。

缺点：

1. 语义漂移风险高，难以保证跨版本可追溯。
2. 无法判定何时可删除 deprecated 值。
3. 不足以支持 T013 checklist。

### 3.2 方案 B：强语义枚举生命周期规则

设计思路：定义默认值、新增值、弃用、删除的完整生命周期和评审条件。

优点：

1. 直接满足 T012 Done Criteria。
2. 与 T002/T003 兼容性规则完全对齐。
3. 可直接转为 T013 可执行检查项。

缺点：

1. 需要团队统一执行 deprecate 元数据记录。

### 3.3 决策

采用方案 B。

取舍理由：方案 B 在不跨任务的前提下，提供最小但完整的枚举演进治理闭环。

## 4. 最终产出

### 4.1 默认值规则

1. 每个可演进枚举必须保留 `Unspecified`（或语义等价 `Unknown`）值。
2. `Unspecified` 必须作为缺省兜底语义，不承载业务成功/失败语义。
3. 任何版本不得删除 `Unspecified`，否则视为 breaking。
4. 新增枚举值时，旧消费方无法识别该值时应降级为 `Unspecified` 处理路径。

### 4.2 命名与语义规则

1. 枚举值命名使用稳定语义词，不使用临时实现细节词。
2. 既有枚举值语义禁止重解释；若语义变化，必须新增值并废弃旧值。
3. 同一枚举内不允许语义重复值（同义不同名）。

### 4.3 deprecate 方式（生命周期）

枚举值生命周期定义：

1. `ACTIVE`：正常可用。
2. `DEPRECATED`：可读可写受限，不建议新增使用。
3. `REMOVED`：仅在满足迁移前提后允许移除（默认后续大版本）。

deprecate 最小记录要求：

| 字段 | 必填 | 说明 |
|---|---|---|
| enum_name | 是 | 枚举类型名 |
| enum_value | 是 | 被弃用的枚举值 |
| status | 是 | DEPRECATED |
| deprecated_since | 是 | 首次标记弃用版本/日期 |
| replacement | 否 | 替代枚举值或替代策略 |
| reason | 是 | 弃用原因 |
| removal_condition | 是 | 何时允许删除（迁移完成、无消费方等） |

deprecate 行为规则：

1. 标记为 DEPRECATED 后，旧值必须仍可被解析与回放。
2. 业务写入路径应停止产生 DEPRECATED 值，读取路径必须兼容。
3. 删除 DEPRECATED 值前必须完成消费方迁移评审。

### 4.4 兼容性判定规则

| 变更动作 | 判定 | 说明 |
|---|---|---|
| 新增枚举值（保留 Unspecified） | review-required | 需验证旧消费方未知值降级策略 |
| 标记枚举值 DEPRECATED | review-required | 非破坏但需迁移计划 |
| 删除已弃用枚举值 | breaking | 需版本迁移与专门评审 |
| 重解释既有枚举值语义 | breaking | 即使名称不变也算 breaking |
| 删除 Unspecified 值 | breaking | 破坏默认兜底语义 |

### 4.5 评审检查项（供 T013 直接消费）

1. 枚举是否保留 Unspecified 值。
2. 新增值是否提供未知值降级策略。
3. deprecate 记录是否包含 replacement、reason、removal_condition。
4. 是否存在语义重解释或直接删除旧值行为。
5. 是否明确读取兼容与写入收敛策略。

### 4.6 Mode Extension

Design 模式产出：

1. 本文档为评审材料，不改代码。
2. 已输出枚举默认值语义、弃用流程、边界与兼容性建议。
3. 可直接落盘路径：docs/todos/contracts/deliverables/WP02-T012-枚举规范.md。

Build 模式预备清单（不在本任务执行）：

1. 候选改动文件：contracts 中各枚举定义头文件与解析层。
2. 关键接口：枚举解析器、兼容降级映射器、弃用告警输出。
3. 冲突检查：若现有代码直接删除旧值，先恢复兼容读取再推进迁移。
4. 验证步骤：
   - 构建：执行 CMake 构建验证 contracts 相关目标。
   - 测试：覆盖 unknown->Unspecified 降级、deprecated 读写行为、删除门禁。
   - 契约校验：验证每个枚举均含 Unspecified 且 deprecate 元数据完整。

1. 已形成枚举规范文档。
2. 已明确“每个枚举保留 Unspecified 值”。
3. 已定义 deprecate 方式与最小元数据。
4. 已明确新增、弃用、删除、重解释的兼容性判定。
5. 产出可直接作为 T013 checklist 输入。

Quality Gate 回答：

1. 当前任务是否达成 Done Criteria：是。
2. 产出是否可被下一任务直接消费：是，可直接被 T013 消费。
3. 是否引入 breaking change 风险：当前仅文档冻结，无实现变更；后续若删除 Unspecified 或重解释旧值会有 breaking 风险。

## 6. 风险与回退

1. 风险：团队把 DEPRECATED 当作可立即删除。
   回退：强制 removal_condition 与迁移评审门禁，未满足前不得删除。
2. 风险：未知值未降级到 Unspecified，导致运行时分支异常。
   回退：统一解析层降级策略并增加契约测试。
3. 风险：同义新值替换旧值但未记录替代关系。
   回退：回滚替换并补齐 replacement/reason 元数据后再推进。
