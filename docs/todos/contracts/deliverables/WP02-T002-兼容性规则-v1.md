## 1. 任务理解

本任务目标是为 WP-02 输出一份可评审、可执行、可回溯的兼容性总体原则草案，作为后续字段演进、枚举规则、错误码规则、事件封套规则和版本冻结说明的上位约束。

本任务只冻结兼容性判断原则、变更分类口径、废弃路径和版本治理门禁，不下钻字段级规则表，不定义具体字段、枚举、错误码或序列化框架策略。

本任务完成后，WP02-T003、T012、T013 以及后续对象冻结任务必须以本规则作为兼容性判断基线。

## 2. 约束与边界

### 2.1 直接约束

1. 来自计划文档第 10.1 节：向后兼容优先于结构清爽；新增字段优于修改旧字段；废弃优于删除；改语义必须视为重大变更，即使字段名不变。
2. 来自计划文档第 10.4 节：contracts 采用语义版本号管理；每次对象冻结都要记录版本变更说明；进入实现消费期后，breaking change 必须经过专门评审并更新迁移说明。
3. 来自计划文档第 11 节和第 12 节：兼容性和版本策略必须可执行；若兼容性策略缺失，会导致实现与测试桩频繁返工，因此应先定义版本与变更规则，再推进对象冻结。
4. 来自 WP-02 TODO：本任务完成判定必须覆盖“新增优于修改、废弃优于删除、语义变化视为 breaking”。
5. 来自 WP02-T001：横切基础对象范围已冻结为 error、event、budget、id、time；本任务不能新增新的横切类别，也不能越权定义各类别字段细节。

### 2.2 本任务边界

1. 只定义兼容性总体原则，不展开字段演进明细表。
2. 只定义 breaking 与 non-breaking 的判断口径，不定义版本号具体升位流程细节。
3. 只定义废弃治理原则，不定义废弃字段的逐项清单。
4. 不改写 ADR-006、ADR-007、ADR-008 已冻结边界。
5. 不提前替代 T003 的字段演进规则、T012 的枚举弃用规则、T013 的正式 review checklist。

### 2.3 非目标

1. 不定义字段类型、可选性、多值性变化的逐条判定表。
2. 不定义枚举默认值和弃用方式的最终规范。
3. 不定义 contract test 的具体用例。
4. 不定义序列化格式、IDL 或跨进程协议版本协商机制。
5. 不生成 contracts 代码或头文件。

## 3. 方案对比与决策

### 3.1 方案 A：最小原则集

仅保留三条 Done Criteria 对应原则：

1. 新增优于修改。
2. 废弃优于删除。
3. 语义变化视为 breaking。

优点：

1. 范围最小，和 TODO 完成判定直接一致。
2. 评审成本低。

缺点：

1. 缺少版本说明、迁移说明和评审门禁，执行时容易落空。
2. 不能直接指导后续对象冻结记录“什么变了、为什么不兼容、如何过渡”。

### 3.2 方案 B：兼容性治理最小闭环

在三条核心原则基础上，增加三类配套规则：

1. 变更分类口径：区分 non-breaking、review-required、breaking。
2. 废弃路径：先 deprecated，再迁移，再删除。
3. 版本治理门禁：对象冻结需附兼容性说明，breaking 需专门评审。

优点：

1. 与计划文档第 10 节和第 12 节完全对齐，能落地执行。
2. 可直接被 T003、T012、T013 和后续冻结包消费。
3. 不需要提前展开字段细则，仍然保持任务原子性。

缺点：

1. 比方案 A 多出治理规则，需要在文档中更明确区分“总体原则”和“后续细则”。

### 3.3 决策

选择方案 B。

决策理由：

1. 方案 B 仍然没有跨出 T002 范围，因为它只定义兼容性治理闭环，不定义字段细则。
2. 方案 B 能支撑计划文档要求的“每次对象冻结都记录版本变更说明”和“breaking change 必须经过专门评审”。
3. 方案 B 更适合作为后续 review checklist 和冻结包模板的上位输入。

## 4. 最终产出

### 4.1 兼容性规则 v1

#### 4.1.1 总体原则

1. 向后兼容优先于结构清爽。
2. 新增优于修改：已有消费方已依赖的对象、字段、枚举和值语义，默认通过新增扩展承接新需求，而不是直接改写已有定义。
3. 废弃优于删除：不再推荐使用的对象或字段，应先进入 deprecated 状态并给出迁移路径，而不是直接移除。
4. 语义变化视为 breaking：只要消费方对同名对象、字段、枚举值的理解需要改变，即使名字未变，也按 breaking change 处理。
5. 兼容性判断优先看消费语义，不优先看实现便利性或文档整洁度。

#### 4.1.2 变更分类规则

本规则将 contracts 变更分为三类：

1. non-breaking：新增可忽略信息，且不会改变既有消费语义与默认行为。
2. review-required：表面不破坏结构，但会影响消费方理解、测试基线或迁移成本，需要在冻结评审中显式确认。
3. breaking：会导致既有消费方语义失效、默认行为改变、解析假设失效或必须同步改代码/测试。

判定口径如下：

1. 新增字段默认按 non-breaking 候选处理，但前提是旧消费方可以安全忽略，且不会因默认值缺省而改变既有语义。
2. 修改既有字段含义、默认解释、成功/失败含义、约束语义，按 breaking 处理。
3. 删除既有字段、删除枚举值、收窄对象职责、改变对象边界，默认按 breaking 处理。
4. 仅增加 deprecated 标记且保持旧语义可用，按 review-required 处理。
5. 影响 ADR 承接边界的任何变化，默认按 breaking 处理并停止在本工作包内扩展。

#### 4.1.3 废弃治理规则

1. 废弃不是立即删除，而是显式进入“保留但不鼓励新增依赖”的过渡状态。
2. 每个废弃项至少应附带三类说明：废弃原因、替代路径、预计移除前提。
3. 在进入实现消费期前，可以登记废弃但不应立即删除，除非尚无任何消费方和测试基线依赖。
4. 已发布到冻结包的对象或字段，若无显式评审结论，不得在后续任务中私自删除。

#### 4.1.4 版本治理规则

1. contracts 按 semantic intent 区分 breaking 与 non-breaking 变更；本任务先冻结分类口径，不细化版本号升位流程。
2. 每次对象冻结都必须记录“本次变更是否 breaking、影响哪些消费方、是否需要迁移说明”。
3. 一旦进入实现消费期，breaking change 必须经过专门评审，并补充迁移说明与影响清单。
4. 若变更无法明确判定是否 breaking，则默认提升为 review-required，不按“安全无影响”处理。

#### 4.1.5 评审门禁规则

后续对象冻结评审至少回答以下问题：

1. 这是新增，还是在改写旧语义。
2. 若保留旧对象，能否通过新增字段或新增对象完成扩展。
3. 若声明废弃，替代路径和迁移窗口是否已经写明。
4. 是否改变了既有消费方对 success、failure、budget、id、time、event 的解释。
5. 是否触发 ADR 边界回退或版本说明更新。

## 6. 风险与回退

### 6.1 主要风险

1. 风险：团队把“新增字段默认 non-breaking”误解为所有新增都天然安全。
   应对：本规则明确要求新增必须可被旧消费方安全忽略，且不得改变既有语义。
2. 风险：后续任务把 review-required 误降级为 non-breaking，绕开正式评审。
   应对：T013 必须把“无法明确判定时默认升级审查”写入 checklist。
3. 风险：废弃规则只停留在口头约定，没有迁移说明，最终仍演变为隐式删除。
   应对：所有废弃项必须附替代路径和预计移除前提，未满足前不得删除。

### 6.2 回退策略

1. 若后续评审认为本规则过宽或过窄，不直接修改已冻结对象定义，先在 T013/T014 中登记争议并形成补充决议。
2. 若发现某条兼容性规则与既有 ADR 边界冲突，以 ADR 结论为准，撤回冲突解释，但保留“兼容优先”的总原则。
3. 若进入实现消费期前发现规则无法支撑真实消费方需求，可在不破坏三条核心原则的前提下发布 v1.x 补充说明，不回退到删除优先或改语义默认允许的口径。
