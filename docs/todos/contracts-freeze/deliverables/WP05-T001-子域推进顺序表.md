### 2.3 对本任务的可落地启发

1. T001 的核心不是把“顺序建议”复制一遍，而是把“哪些可以并行、哪些必须等待前一波冻结完成”压成可执行 guard。
2. tool 需要单独作为 Wave 1，因为它直接影响 Observation 折叠和治理链，后续多个子域都会消费这条语义通道。
3. prompt 和 memory 可以作为同一波次并行推进：两者都只消费 WP03/WP04 已冻结对象，不互相回改对象定义，但都必须在 tool 稳定后启动。
4. task 和 event 可以作为下一波并行推进：task 复用 WP04 协同边界，event 复用 WP02 EventEnvelope，二者互不占有对方语义，但都不应在 Wave 2 尚未稳定前启动。
5. llm 必须最后收口，因为它最接近 provider 适配层；如果过早推进，最容易把 provider 私有字段反向污染前面已经冻结的共享对象。

## 4. WP05 Rollout 波次定义

### 4.1 总体波次

| Wave | 子域 | 并行策略 | 启动前提 | 本波完成后可进入 |
|---|---|---|---|---|
| Wave 1 | tool | 单独推进 | WP04 M4 冻结包已发布 | Wave 2 |
| Wave 2 | prompt + memory | 允许并行 | tool 完成 | Wave 3 |
| Wave 3 | task + event | 允许并行 | prompt、memory 完成 | Wave 4 |
| Wave 4 | llm | 单独推进 | task、event 完成 | 接口准入链与测试治理链 |

### 4.2 排序理由

1. tool 先行：ToolRequest/ToolResult/ToolDescriptor/ToolIR 会影响 Observation 折叠、工具治理和后续可见能力描述，属于高扇出上游子域。
2. prompt 与 memory 并行：PromptSpec/PromptRelease 主要承接 WP04 Prompt 边界；Turn/Session/SummaryMemory/MemoryFact 主要承接 WP03 ContextPacket/Observation/Checkpoint。二者消费同一冻结骨架，但不互相定义对方语义。
3. task 与 event 并行：task 继续细化协同执行单元；event 继续细化 EventEnvelope 之上的事件类型与 payload。两者都应建立在前两波稳定后，以免重新发明主链路或共享上下文语义。
4. llm 最后：LLMRequest/LLMResponse 紧贴 provider 适配层，应在 prompt、memory、task、event 的共享对象冻结后再进入，避免 provider 私有字段渗透到共享 contracts。

## 5. 并行边界与越权禁区

### 5.1 允许并行的边界

1. prompt 与 memory 允许同波并行，但只能在 tool 完成后启动。
2. task 与 event 允许同波并行，但只能在 prompt 与 memory 都完成后启动。

### 5.2 明确禁止的并行关系

1. tool 不能与任何其他子域并行启动。
2. Wave 2 期间，不允许 task、event、llm 提前启动。
3. Wave 3 期间，不允许 llm 提前启动。
4. 同一子域不允许重复启动或在“已完成”后再次进入 rollout。

### 5.3 越权禁区

| 子域 | 不得回改/吸收的已冻结语义 |
|---|---|
| tool | 不重复定义 ErrorInfo、RuntimeBudget、Observation 主链路折叠语义 |
| prompt | 不回改 ContextPacket/PromptComposeRequest/PromptComposeResult 的边界 |
| memory | 不回改 Observation、ContextPacket、Checkpoint 的主链路语义 |
| task | 不承担顶层 Session/FSM/最终输出主控语义 |
| event | 不把 EventEnvelope 基础封套回退成 payload 专属 schema |
| llm | 不把 provider 私有字段泄漏进共享主链路对象 |
