## 3. 方案对比与决策

### 3.1 方案 A：直接发布无条件冻结包

优点：

1. 交付速度快。
2. 可立即宣告 M2 通过。

缺点：

1. 与 T014 遗留项 L1/L2 冲突，缺乏风控闭环。
2. 可能把潜在 breaking 风险扩散到 WP-03/WP-04。

### 3.2 方案 B：有条件冻结发布（推荐）

优点：

1. 保持 M2 主语义冻结，同时显式管理未闭环风险。
2. 与 T014 结论一致，可追溯且可审计。
3. 后续子域有明确准入门禁，避免再发明基础语义。

缺点：

1. 需要维护遗留项关闭状态。
2. 发布结论为 CONDITIONAL FREEZE，不是绝对完成态。

### 3.3 决策

采用方案 B：发布有条件生效的 M2 冻结包。

取舍理由：满足“发布冻结版”目标，同时遵守“前置依赖存在遗留时必须显式阻塞管理”的约束。

### 4.3 冻结语义主结论

1. 错误语义冻结：ResultCode 与 ErrorInfo/ErrorSource 作为统一失败表达，后续子域禁止重定义同类主语义。
2. 预算语义冻结：RuntimeBudget 与 BudgetSnapshot 为唯一横切预算表达，禁止子域自行发明 token/turn/tool_call/latency/replan 顶层语义。
3. 标识与时间冻结：request_id、session_id、trace_id、task_id、lease_id 及 created_at/deadline_at/timeout_ms/ttl 规则统一，后续仅允许兼容扩展。
4. 事件封套冻结：EventEnvelope 头部只承载通用字段，模块私有字段必须留在 payload。
5. 枚举演进冻结：每个枚举保留 Unspecified，弃用优先于删除。

### 4.4 生效门禁（M2 Gate）

进入 WP-03/WP-04 前必须满足：

1. 新对象评审必须逐项通过 WP02-T013 Checklist。
2. 任一子域若引入新的横切错误/预算/标识/时间/封套主语义，直接判定为 REJECT。
3. [WP02-T015-B1-timeout迁移清单](WP02-T015-B1-timeout%E8%BF%81%E7%A7%BB%E6%B8%85%E5%8D%95.md) 已落盘，可作为秒到毫秒迁移评审依据。
4. [WP02-T015-B2-枚举降级契约测试基线](WP02-T015-B2-%E6%9E%9A%E4%B8%BE%E9%99%8D%E7%BA%A7%E5%A5%91%E7%BA%A6%E6%B5%8B%E8%AF%95%E5%9F%BA%E7%BA%BF.md) 已落盘，并由 `dasall_contract_compatibility_test` 作为待执行证据。
