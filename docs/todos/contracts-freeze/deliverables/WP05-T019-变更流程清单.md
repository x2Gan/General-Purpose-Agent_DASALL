## 1. contracts 变更评审流程清单（V1）

### 1.1 入口

1. 先执行 `validate_version_change_schema`，确认模板字段和 semver 合法。
2. 判断是否 breaking：`version_change_is_breaking`。

### 1.2 流程分支

1. 非 breaking：
- 结果标记 `NotRequired`，允许放行。

2. breaking：
- 必须满足 `breaking_review_required=true`。
- 必须有 `owner_approved=true`。
- 必须有 `consumer_approved=true`。
- 必须提供迁移验证证据（`migration_validation_evidence` 非空）。
- 必须提供评审纪要链接（`review_record_link` 非空）。

### 1.3 出口

1. `Approved`：breaking 且所有门禁通过。
2. `Rejected`：breaking 但至少一项门禁失败。
3. `NotRequired`：非 breaking，不触发专项评审。

## 2. Design 原子清单

1. D1：定义 breaking 评审上下文和结果模型。
- 输入依据：T018 schema 与 guard。
- 产出路径：`BreakingReviewGuards.h`。
- 完成判定：可表达 NotRequired/Approved/Rejected 三态。
- 风险与回退：若语义冲突，优先保持 non-breaking 直通、breaking 强校验。

2. D2：定义 breaking 审批门禁规则。
- 输入依据：1.2 流程分支。
- 产出路径：`BreakingReviewGuards.h`。
- 完成判定：可输出首失败检查点与失败原因。
- 风险与回退：若规则过重，保留最小三门禁（owner/consumer/migration evidence）。

3. D3：锁定 Build 三件套与发现性。
- 输入依据：`tests/contract/CMakeLists.txt`。

- 风险与回退：若不可发现，优先修 CMake 注册。
