### 3.2 关键发现（按严重级别）

#### 严重级别：Critical

1. 头文件组合 include 会触发编译失败（同名函数重定义）
- 位置：
  - [contracts/include/boundary/IdentityMetadata.h](../../../contracts/include/boundary/IdentityMetadata.h#L30)
  - [contracts/include/event/EventEnvelopeGuards.h](../../../contracts/include/event/EventEnvelopeGuards.h#L15)
  - [contracts/include/error/ErrorInfoGuards.h](../../../contracts/include/error/ErrorInfoGuards.h#L14)
  - [contracts/include/error/ErrorSourceGuards.h](../../../contracts/include/error/ErrorSourceGuards.h#L15)
- 现象：
  - has_non_empty_value 在两个头中同签名重定义。
  - is_supported_error_source_ref_type 在两个头中同签名重定义。
- 复现命令：
  - c++ -std=c++17 -Icontracts/include -c /tmp/dup_check.cpp -o /tmp/dup_check.o
- 结论：阻断合并（Changes Requested）。
- 最小修复建议：
  - 将上述 helper 下沉到公共头（如 boundary/CommonGuards.h 或 error/ErrorSourceCommon.h），各处仅声明/复用一次。
  - 或改为文件内私有命名空间并统一唯一命名，避免同名全局 inline 函数。

#### 严重级别：Major

2. timeout_seconds 到 timeout_ms 迁移缺少上溢保护
- 位置：
  - [contracts/include/boundary/CompatibilityGuards.h](../../../contracts/include/boundary/CompatibilityGuards.h#L33)
- 问题：
  - timeout_seconds * 1000 后直接转换为 uint32_t，超大输入可能溢出回绕。
- 风险：
  - 可能生成错误 deadline，进而影响预算与超时判定。
- 修复建议：
  - 在乘法前做上界检查：timeout_seconds <= UINT32_MAX / 1000。
  - 失败时返回 ok=false 并给出 reason（如 timeout_seconds overflow）。
  - 增加对应负例测试。

3. BudgetSnapshot remaining 计算存在大数转换风险
- 位置：
  - [contracts/include/checkpoint/BudgetSnapshotGuards.h](../../../contracts/include/checkpoint/BudgetSnapshotGuards.h#L55)
- 问题：
  - 将 uint64_t max/current 转 int64_t 再相减，超 INT64_MAX 时有实现相关行为风险。
- 风险：
  - 边界输入下比较结果不可靠，可能误判 over budget。
- 修复建议：
  - 使用无符号分支计算：
    - current <= max 时 remaining = max-current（非负）
    - current > max 时 remaining = -(current-max)（先在 uint64 计算差值，再做有界转换）
  - 增加极值测试。
