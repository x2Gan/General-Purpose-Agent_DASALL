## 0. Phase 0 研究学习证据链

### 本地证据清单

1. `docs/plans/DASALL_contracts冻结实施计划.md` 阶段 5 已冻结准入前提：只有真正跨模块且需要作为稳定依赖面的抽象接口，才进入 contracts；数据对象稳定前，不设计 I 开头接口。
2. `docs/todos/contracts-freeze/deliverables/WP05-T011-接口候选清单.md` 已将候选集收敛为 10 个跨模块接口，并显式区分 `ReviewReady` 与 `AwaitingSupportingContracts`；这意味着 T012 的工作是“在候选集中做准入/推迟/退回”，而不是重新发明候选目录。
3. `docs/architecture/DASSALL_Agent_architecture.md` 第 7.4 节已冻结依赖方向：runtime 依赖 llm/tools/memory/services 抽象，cognition 只能依赖 contracts 与抽象接口，因此 admission 守卫必须继续维持跨模块依赖倒置，不允许同模块内部接口混入共享 contracts。
4. `docs/architecture/DASSALL_Agent_architecture.md` 第 7.3 节当前只有 `IToolManager` 与 `ILLMAdapter` 已具备与既有冻结对象直接闭合的输入输出 handoff；其余接口仍引用 `PlanGraph`、`SessionSnapshot`、`RetrievalQuery` 等未冻结 supporting contracts。
5. ADR-006 与 ADR-008 已分别固定 `IContextOrchestrator`、`IAgentRegistry`、`IResultMerger` 的职责方向，但并未提供可直接落入 contracts 的完整 supporting object 集，因此这些接口当前只能推迟，不能直接准入。

### 外部参考清单

1. Microsoft Azure Architecture Center, Strangler Fig pattern：稳定门面应先于内部替换落稳，迁移要以分阶段、低风险方式推进；可直接映射为“成熟依赖面先准入，不成熟依赖面先推迟”。
2. Protocol Buffers Best Practices：客户端与服务端升级不会同步，共享契约必须优先兼容演进，不能把仍在快速变化的 supporting contracts 过早暴露成共享接口。

### 对本任务的可落地启发

1. 准入守卫应当只基于 T011 已冻结的目录元数据进行二值/三值判定，而不是试图解析真实接口签名。
2. `ReviewReady` 应直接映射为 `Admit`，`AwaitingSupportingContracts` 应直接映射为 `Postpone`，目录外接口应统一 `Return`。
3. 除 readiness 外，守卫还必须检查 catalog 元数据是否完整，避免“目录里有名字但没有 anchor/rationale”这种伪准入。
4. admission 测试必须至少包含：准入正例、推迟负例、目录外退回负例，以及元数据不完整/同模块伪依赖的守卫负例。

## 2. 准入结论

### 2.1 Admit

1. `IToolManager`
2. `ILLMAdapter`

结论依据：

1. 均已存在明确架构草图。
2. 均已具备冻结的请求/响应或目录对象族作为 stable anchor。
3. 可直接进入共享 contracts admission review，不需要再等待 supporting contracts。

### 2.2 Postpone

1. `IPlanner`
2. `IMemoryStore`
3. `IContextOrchestrator`
4. `IKnowledgeService`
5. `IExecutionService`
6. `IDataService`
7. `IAgentRegistry`
8. `IResultMerger`

结论依据：

1. 这些接口具备跨模块稳定依赖面的潜力，因此不应退回。
2. 但它们仍缺少完整冻结的 supporting contracts，若当前直接准入会把未成熟对象面提前固化到共享 contracts。

### 2.3 Return

1. 任何不在 `InterfaceCatalog` 中的接口名。
2. 任何 catalog 元数据不完整的条目。
3. 任何 owner 与 primary consumer 未形成跨模块边界的条目。

结论依据：

1. 目录外接口意味着未通过 T011 候选筛选，应回到模块内部继续澄清。
2. 元数据不完整会破坏 T012 的可追溯审查性。
3. 同模块依赖不构成共享 contracts 准入面。

## 3. Design 原子清单

1. D1：把 T012 结论压缩成稳定 decision 集。
- 输入依据：T011 候选目录与阶段 5 准入原则。
- 产出路径：本文件第 2 节、`InterfaceAdmissionGuards.h`。
- 完成判定：decision 仅包含 `Admit`、`Postpone`、`Return` 三类，且原因稳定可断言。
- 风险与回退：若 decision 被扩张成评审流程 DSL，则回退到最小 decision 集。

2. D2：定义 admission 的最小程序化规则。
- 输入依据：T011 catalog 字段、架构 7.4 跨模块依赖规则。
- 产出路径：`contracts/include/boundary/InterfaceAdmissionGuards.h`。
- 完成判定：守卫至少检查 catalog presence、metadata completeness、cross-module boundary、readiness。
- 风险与回退：若守卫开始依赖真实接口签名，则回退到目录元数据输入。

3. D3：锁定可验证的 admission 负例矩阵。
- 输入依据：Build 三件套要求。
- 产出路径：`tests/contract/smoke/InterfaceAdmissionContractTest.cpp`。
- 完成判定：至少覆盖准入正例、推迟负例、目录外退回负例、元数据不完整退回负例、同模块伪依赖退回负例。
- 风险与回退：若只有 happy path，则补齐上述负例。
