## 1. 任务理解

本任务目标是把 WP02-T002 的兼容性总体原则，下钻为可直接用于对象字段设计和评审的字段演进规则表，供后续 error、budget、id、time、event 及主链路对象字段冻结时统一使用。

本任务只处理字段层面的演进规则，不定义具体对象字段清单，不定义枚举完整规范，不定义错误码或事件头部字段，也不进入代码实现。

本任务完成后，后续对象设计评审必须能够针对字段类型、可选性、多值性和枚举默认值给出一致判定。

## 2. 约束与边界

### 2.1 直接约束

1. 来自计划文档第 10.2 节：不随意修改字段类型；不把可选字段改为强制字段；不把多值字段收窄为单值字段；枚举必须保留未指定值；不使用语义可能持续扩张的布尔字段表达状态机。
2. 来自 WP02-T002：新增优于修改、废弃优于删除、语义变化视为 breaking；字段级变化需要按 non-breaking、review-required、breaking 分类处理。
3. 来自 WP-02 TODO：本任务完成判定必须覆盖字段类型、可选性、多值性、枚举默认值。
4. 来自工程规范：contracts 冻结前不做过度复杂化设计，公共接口优先保持可判定成功/失败语义，字段设计应服务契约一致性而非实现便利。

### 2.2 本任务边界

1. 只定义字段演进规则，不定义对象级字段清单。
2. 只定义字段变化的兼容性判定，不定义版本号升位流程。
3. 只给出枚举默认值的演进口径，不替代 T012 的完整枚举规范与弃用规则。
4. 不改写 ADR-006、ADR-007、ADR-008 的边界语义。
5. 不展开序列化框架、IDL tag、数据库迁移或网络协议兼容方案。

### 2.3 非目标

1. 不定义 ErrorInfo、RuntimeBudget、EventEnvelope 等对象的最终字段集合。
2. 不定义字段命名规范和命名模板。
3. 不定义默认值注入的运行时实现方式。
4. 不定义 contract tests 的详细断言列表。
5. 不生成 contracts 头文件或测试代码。

## 3. 方案对比与决策

### 3.1 方案 A：按字段属性分项规则表

围绕字段类型、可选性、多值性、枚举默认值、布尔状态表达五类属性，给出“允许变化、慎改变化、禁止变化、兼容性判定、推荐替代方案”。

优点：

1. 与计划文档第 10.2 节一一对应，可追溯性强。
2. 能直接用于后续对象评审和 checklist 生成。
3. 不会越权到对象专属字段设计。

缺点：

1. 文档偏规则化，需要额外说明适用方式。

### 3.2 方案 B：按对象生命周期给案例式说明

围绕“初始设计、扩展、废弃、移除”四个阶段，用案例说明字段如何变化。

优点：

1. 可读性更强，易于沟通。
2. 对新评审者更友好。

缺点：

1. 容易遗漏字段属性维度，和第 10.2 节映射不够直接。
2. 更容易引入超出本任务的对象案例和实现假设。

### 3.3 决策

选择方案 A。

决策理由：

1. 方案 A 与 T003 完成判定直接对齐。
2. 方案 A 可作为 T012、T013 的直接输入，不需要二次抽象。
3. 方案 A 更容易避免跨任务扩张范围。

### 4.2 字段演进判定说明

#### 4.2.1 字段类型

1. 既有字段类型一旦被消费方依赖，默认不可直接改写。
2. 若确需表达更丰富语义，应优先新增并行字段或新对象，而不是把旧字段原位升级。
3. 即使新旧类型在某些实现中可互转，只要消费方需要改变解释方式，仍按 breaking 处理。

#### 4.2.2 可选性

1. 新增字段默认应设计为可选，确保旧消费方在未知该字段时仍可工作。
2. 把既有可选字段改为必填，本质上是提高对象合法性门槛，属于 breaking。
3. 若业务上出现“新版本必须提供”的需求，应通过新增对象版本、上层约束或迁移窗口处理，不直接回写旧字段。

#### 4.2.3 多值性

1. 多值到单值的收窄会丢失信息表达能力，默认属于 breaking。
2. 单值到多值虽然表面更宽，但若消费方原本只处理单个元素，可能改变语义，应至少进入 review-required。
3. 需要同时表达“首选值 + 候选集合”时，应拆出显式字段，而不是隐式复用同一字段语义。

#### 4.2.4 枚举默认值

1. 每个可演进枚举都必须保留未指定值，用于旧消费方兜底、缺省输入和未知值兼容。
2. 新增枚举值时，不得重写旧值含义，也不得删除未指定值。
3. 枚举值的语义重解释属于 breaking，即使枚举常量名字不变。

#### 4.2.5 布尔状态字段

1. bool 只适合长期稳定的开关语义，不适合表达阶段、模式、策略决策或状态机分支。
2. 若某字段已出现“今天二值、明天三值”的趋势，应在设计阶段直接转为 enum 或对象组合，避免后续从 bool 迁移造成 breaking。

### 4.3 评审使用方式

后续对象字段评审至少回答以下问题：

1. 这次变更是新增字段，还是在改写旧字段属性。
2. 若改动字段类型、可选性或多值性，是否存在通过新增字段规避 breaking 的方案。
3. 枚举是否保留未指定值，且新增值不会改写旧语义。
4. 是否错误地用 bool 承载了可扩展状态。
5. 若必须 breaking，是否已记录迁移说明并升级到专门评审。

## 6. 风险与回退

### 6.1 主要风险

1. 风险：团队把“新增字段优于修改”机械套用到所有字段，导致并行字段过多、对象膨胀。
   应对：当字段膨胀开始影响对象清晰度时，应按计划文档第 10.3 节优先拆成组合对象，而不是继续堆叠同义字段。
2. 风险：把单值扩多值误判为天然兼容，忽略消费方只支持单元素的事实。
   应对：本规则将单值扩多值至少提升为 review-required，不按默认安全处理。
3. 风险：枚举保留未指定值被形式化执行，但消费方没有定义未知值兜底策略。
   应对：T012 和 T013 必须补充“未知值如何降级处理”的检查项。

### 6.2 回退策略

1. 若后续评审发现某条字段规则过宽或过窄，不直接修改既有对象定义，先在 T013/T014 中登记争议并形成补充决议。
2. 若某对象设计必须违反本规则，应显式标记为 breaking 候选并走专门评审，不允许在对象设计中隐式例外。
3. 若后续发现某字段问题本质上是对象拆分问题，应回退到对象拆分层处理，不在原字段上持续叠补丁。
