# ADR-005 DASALL Agent 架构评审结论与落地基线

## 0. 文档信息

### 0.1 文档定位

本文档是 DASALL Agent 当前阶段的正式架构评审稿，用于沉淀本轮评审结论、明确方案可接受范围、识别阻塞落地的关键问题，并给出后续逐模块详细设计的基线顺序。

本文档不是模块详细设计文档，也不是实现任务拆解清单。本文档的作用是回答三个问题：

1. 当前总体架构方案是否成立。
2. 当前工程骨架与架构蓝图之间的差距在哪里。
3. 后续详细设计应该按什么顺序推进，哪些问题必须先冻结。

### 0.2 评审范围

本轮评审覆盖以下内容：

1. 总体架构设计文档中的分层、职责、边界、运行模型和治理机制。
2. 工程架构蓝图中的目录映射、依赖规则、接口规划、Profile 策略与测试规划。
3. 当前仓库工程骨架与蓝图之间的一致性。
4. 后续逐模块详细设计的推荐推进顺序。

本轮评审不包含以下内容：

1. 外部最新 Agent 方案的专题对标。
2. 具体代码实现方案。
3. 单个模块的详细类图、时序图与接口字段级设计。

### 0.3 评审依据

本轮评审主要基于以下材料：

1. docs/architecture/DASSALL_Agent_architecture.md
2. docs/architecture/DASALL_Engineering_Blueprint.md
3. 当前仓库顶层模块目录、CMakeLists.txt、应用入口、测试骨架与占位实现

---

## 1. 评审结论摘要

### 1.1 总体结论

当前 DASALL Agent 的总体方案在架构方向上成立，可以作为后续工程化落地的基础蓝图继续推进。

该方案在以下方面具备明显优点：

1. 采用了清晰的分层结构，能够支撑长期演进。
2. 明确强调控制与认知分离、认知与执行分离，符合高风险执行型 Agent 的工程规律。
3. 将契约、治理、恢复、审计、Profile 裁剪纳入主架构，而不是事后补丁。
4. 对嵌入式与边缘场景下的资源约束、平台差异、长期运行问题有明确考虑。

但是，当前方案还不能直接进入大规模并行编码阶段。原因不是总体设计错误，而是以下前置条件尚未满足：

1. contracts 层尚未冻结，跨模块对象和接口边界还没有真正落到代码与文档统一基线。
2. 若干关键组件的职责边界仍存在重叠或不明确，继续展开详细设计会产生返工。
3. 当前仓库实现状态仍处于脚手架阶段，不具备验证总体方案闭环正确性的最小运行链路。

### 1.2 评审裁定

本轮评审给出的正式裁定如下：

1. 架构方向通过。
2. 工程落地方式附条件通过。
3. 在 contracts、运行主控边界、上下文装配边界、失败处理边界冻结之前，不建议各模块并行展开详细设计和编码。

## 2. 架构方案的主要优点

### 2.1 分层合理，适合复杂 Agent 系统工程化落地

从 apps、runtime、cognition、execution and collaboration、services、platform、infra 的七层划分来看，当前方案没有把 LLM 直接放在系统控制中心，而是把 runtime 作为真正的控制平面。这一点是正确的。

对于嵌入式和边缘系统而言，必须由可控、可审计、可恢复的主控链路管理状态机、预算、超时、降级和恢复，而不是由模型输出直接驱动副作用执行。当前方案在这方面判断准确。

### 2.2 治理链路完整，符合高风险执行型 Agent 的安全要求

工具链路设计为 ActionDecision 到 ToolRoute 到 Validator 到 PolicyGate 到 Executor 到 Audit 到 ObservationDigest，这条链路具备完整的治理语义。对于后续可能接入外部执行控制、文件写入、设备控制、流程触发的场景，这是必要设计，不是可选优化。

### 2.3 Profile 设计兼顾平台裁剪与运行治理

Profile 不是简单的编译开关，而是同时包含 enabled modules、runtime budget、model route、token budget policy、timeout policy、execution policy、ops policy。这个设计比传统嵌入式项目中只做编译宏裁剪更成熟，适合同一主流程面向桌面、边缘和产测形态复用。

### 2.4 Memory、Knowledge、Tools、Multi-Agent 被设计为独立执行协同层

当前方案没有把记忆、知识、工具、多 Agent 混在 cognition 中，而是把它们置于独立协同层，由 runtime 统一编排。这有利于未来替换实现、做降级、做测试隔离，也更符合实际工程中“推理”和“能力执行”分层的要求。

### 2.5 非功能需求不是附属品，而是主架构组成部分

日志、审计、监控、配置、健康检查、OTA 在蓝图中不是后置章节，而是被明确放入 infra 层，并在依赖规则中要求所有层单向依赖。这个做法是正确的，否则系统一旦进入长期运行阶段，后期补 observability 和运维治理的成本会非常高。

---

## 3. 关键问题与风险

### 3.1 当前最大的阻塞不是实现不足，而是契约和边界还未冻结

蓝图中列出了较完整的冻结对象与接口清单，但当前 contracts 目录尚未承载这些对象定义。这意味着：

1. 各模块详细设计缺少共同语言。
2. 测试、Mock、实现之间缺少统一依赖面。
3. 后续一旦各模块先行设计，极易出现字段定义漂移、语义不一致和接口反复修改。

对 Agent 这类跨模块对象极多的系统来说，契约未冻结时推进详细设计，通常会把返工成本放大到运行时主链路和测试体系中。

### 3.2 几组关键职责边界存在重叠，若不先裁定会产生架构内耗

当前最需要先裁定的边界有三组：

1. ContextOrchestrator 与 PromptComposer 的边界。
2. ReflectionEngine 与 RecoveryManager 的边界。
3. AgentOrchestrator 与 MultiAgentCoordinator 的边界。

这三组边界分别对应三类常见架构失控点：

1. 上下文装配职责重复，导致上下文裁剪、排序、注入策略散落在多个模块。
2. 失败分析和失败执行分工不清，导致重试、重规划、降级策略重复实现。
3. 主控调度和多 Agent 分派重叠，导致系统出现双控制中心。

如果不先用 ADR 把这三组边界定下来，后续模块设计会看起来都“有道理”，但最后整体会拼不回一个稳定主流程。

### 3.3 当前文档在若干关键语义上还不够收敛

以下几个语义需要继续收敛：

1. Observation 与 ObservationDigest 的适用边界。
2. PolicyGate、PolicyDecision、PromptPolicy 之间的数据与组件关系。
3. Skill 的选择时机、实例化位置、权限约束流转。
4. BeliefState 中“已确认事实”的判定规则。
5. ResumeToken 与 Checkpoint 的最小必需字段。

这些问题不解决，后续模块文档会形成术语表面一致、语义实际不一致的问题。

### 3.4 依赖规则是合理的，但仍需防止实现期出现隐性闭环

当前蓝图的依赖方向原则总体正确，但在实现阶段需要重点防守以下隐性闭环风险：

1. memory 为了做上下文装配而直接感知 prompt 细节。
2. tools 为了增强能力而直接调用 llm。
3. runtime 为了跨模块省事而直接依赖具体实现，而不是接口。
4. multi_agent 在执行层内部形成独立调度闭环，弱化 runtime 主控地位。

这些闭环往往不是在文档阶段出现，而是在工程实现时因“先跑起来再说”逐步引入，必须提前设防。

## 4. 当前工程骨架评估

### 4.1 当前仓库定位

当前仓库应明确定位为架构脚手架，而不是一期可运行实现。

这个判断基于以下事实：

1. apps 层入口存在，但当前仍是占位输出，例如 apps/cli/src/main.cpp 仅输出 placeholder 文本。
2. runtime、cognition、knowledge、services、multi_agent 等模块的 CMake 已建立，但主体仍以 placeholder.cpp 维持空库可编译。
3. runtime/CMakeLists.txt 当前仅编译 src/placeholder.cpp，尚未承载主控逻辑。
4. infra、platform、llm、tools、memory 的目录框架已存在，但主体实现尚未落位。
5. 当前少数具有实际内容的部分主要集中在 tests/mocks 这类测试支撑代码中。

这说明当前仓库的价值主要在于：

1. 验证工程目录结构和构建骨架是否合理。
2. 为后续接口冻结和模块落地提供占位工程空间。
3. 提前建立统一的命名、模块和 CMake 组织方式。

### 4.2 当前状态带来的风险

如果不明确承认当前仍是脚手架阶段，团队会产生两个误判：

1. 误以为系统已经进入实现中后期，导致对进度和风险评估失真。
2. 误以为可以同时启动多个模块实现，最终在 contracts 和主控边界上集中返工。

因此，当前最重要的不是尽快补更多 placeholder 之外的代码，而是先建立一套真正可持续的设计冻结顺序。

## 5. 正式评审决定

### 5.1 允许继续推进的内容

以下内容可以立即继续推进：

1. contracts 核心对象与 IXxx 接口冻结。
2. runtime 主循环、状态机、调度权边界的详细设计。
3. 关键边界问题的补充 ADR。
4. 最小可运行链路的测试基线设计。

### 5.2 暂不建议并行展开的内容

以下内容暂不建议全面并行展开：

1. 各业务模块基于假定接口直接开始编码。
2. 多 Agent、MCP、RAG、经验记忆等高耦合高级能力的完整实现。
3. 大规模平台适配和性能优化。

原因很简单：主控边界、契约对象和失败治理语义还未完全冻结，此时并行铺开会将问题扩散到更多模块。

### 5.3 必须先补充的三份 ADR

建议在进入模块详细设计前，先新增以下三份 ADR：

1. ContextOrchestrator 与 PromptComposer 的职责划分。
2. ReflectionEngine 与 RecoveryManager 的职责划分。
3. AgentOrchestrator 与 MultiAgentCoordinator 的职责划分。

这三份 ADR 的作用不是补充概念，而是把系统中最容易形成双重控制面的地方先钉死。

## 6. 后续逐模块详细设计顺序

建议后续详细设计按以下顺序推进：

1. contracts
2. runtime
3. cognition
4. llm
5. tools
6. memory
7. knowledge
8. services
9. platform
10. infra
11. multi_agent
12. apps 与 tests 收口

### 6.1 顺序说明

先做 contracts，是因为系统需要统一的数据对象、错误语义、检查点语义和跨模块接口。

先做 runtime，是因为它是系统主控，不先定义状态机、调度、预算、恢复和生命周期管理，其余模块只能各自假设调用时机。

先做 cognition 再做 llm，是因为必须先界定认知层需要什么能力，再决定 LLM 子系统如何提供统一语义。

tools、memory、knowledge 都是 runtime 调度下的协同子系统，应在主控与认知边界清晰后展开。

services、platform、infra 更偏实现承载层，应在上层对象和接口需求明确后再细化，否则很容易提前抽象过度。

multi_agent 应放在单 Agent 主流程稳定之后推进，否则会把尚未收敛的单体控制问题放大到分布式协同问题。

---

## 7. 下一阶段的准入标准

只有满足以下条件，才建议进入多模块详细设计并行阶段：

1. contracts 中核心对象和接口完成首轮冻结。
2. runtime 的状态机、调度模型、预算模型、失败收敛路径完成文档级裁定。
3. 上述三份关键边界 ADR 完成并通过评审。
4. 最小可运行链路完成设计：CLI 到 runtime 到 cognition stub 到 llm mock 或 tool mock 到 response。
5. contract tests 与 mocks 的最小基线明确可执行。

如果这些条件未满足，就不建议把精力投入到大规模功能实现上。

---

## 8. 评审后的执行建议

### 8.1 近期建议

1. 先完成 contracts 详细设计，而不是先做某个功能模块的实现。
2. 立即补三份边界 ADR，避免后续文档继续散开。
3. 定义最小可运行链路，并以此作为后续设计验证主线。

### 8.2 中期建议

1. 在 runtime 详细设计中，把状态机、事件模型、恢复模型、超时模型一次性设计清楚。
2. 在 llm 和 tools 详细设计中，严格区分语义决策与外部执行。
3. 在 memory 设计中，先收敛 ContextPacket、SummaryMemory、ExperienceMemory 写回语义，再做存储选型细节。

### 8.3 风险提示

如果团队跳过 contracts 和 runtime 直接分头细化各模块，最终最有可能出现的问题不是代码质量差，而是：

1. 模块都能自圆其说，但拼不成统一主流程。
2. 同一概念在不同模块中出现多个版本。
3. 错误恢复、权限治理、上下文治理在多个模块重复实现。
4. 测试无法建立稳定 Mock 面，导致接口频繁变化。

---

## 9. 最终裁定

本轮正式评审的最终结论如下：

1. DASALL Agent 的总体架构方向正确，具备继续推进价值。
2. 当前蓝图质量高于当前实现状态，仓库仍处于脚手架阶段。
3. 下一步工作的重点不是立即扩写实现，而是先冻结 contracts 和主控边界。
4. 后续逐模块详细设计可以继续，但必须遵循先契约、再主控、后协同、最后承载与入口的顺序。

Status：Accepted with conditions
