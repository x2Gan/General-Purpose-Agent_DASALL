# DASALL 子系统专项 TODO 生成提示词模板

文档版本：v1.0  
日期：2026-03-24  
状态：Active  
适用范围：依据子系统或子模块详细设计文档，生成可直接进入执行与评审的专项 TODO 文档

---

## 1. 目标与定位

本模板用于把单个子系统或子模块的详细设计文档，转换为一份可执行、可验证、可追溯的专项 TODO 文档。

目标不是重复详细设计，而是把设计结论压缩为可落盘的工程任务，并满足以下要求：

1. 每个任务都能追溯到详细设计中的明确章节、接口语义或对象定义。
2. 每个任务都必须包含代码目标、测试目标、验收命令三件套。
3. 默认把任务拆到接口、函数、数据结构、适配器、测试用例级别。
4. 若输入证据不足以支撑函数级拆分，必须显式降级到类级或组件级，并说明原因。
5. 必须输出 Design -> TODO 映射、阻塞项、解阻条件、执行顺序与质量门。

非目标：

1. 不改写详细设计结论。
2. 不替代 Build 执行模板。
3. 不跨多个子系统合并生成“大而全”总表。
4. 不在输入不足时臆造函数名、字段名或测试出口。

---

## 2. 使用方式

1. 先替换第 3 节输入占位符。
2. 复制第 4 节主提示词作为执行入口。
3. 让执行代理先做“粒度可行性评估”，再生成 TODO。
4. 对输出结果使用第 7 节质量门做审查。
5. 审查通过后，再将专项 TODO 作为后续执行模板或 implementation cycle 的输入。

建议使用节奏：

1. 一次只处理一个子系统或一个子模块。
2. 优先基于 Detailed Design 文档生成 TODO，不直接从架构蓝图跳到编码任务。
3. 若详细设计缺少接口语义、对象字段、异常语义或测试出口，先补设计，再生成 TODO。

---

## 3. 输入占位符

- <PROJECT_ROOT>：项目根目录
- <DATE>：当前日期
- <SUBSYSTEM_NAME>：子系统或子模块名称，例如 infra、infra/logging、runtime/scheduler
- <OUTPUT_TODO_PATH>：专项 TODO 输出路径
- <DETAIL_DESIGN_DOC>：详细设计文档路径
- <ARCH_DOCS>：架构文档路径列表
- <BLUEPRINT_DOCS>：工程蓝图路径列表
- <ADR_DOCS>：相关 ADR 路径列表
- <PLAN_DOCS>：实施计划、阶段计划、冻结计划路径列表
- <DEV_RULES_DOC>：工程协作与编码规范路径
- <CODE_PATHS>：当前子系统代码、测试、CMake 路径列表
- <EXISTING_TODOS>：现有 TODO、交付记录、工作日志路径列表
- <EXTERNAL_REFERENCE_HINTS>：可选，行业参考方向

### 4.4 Task

请设计子系统工程化计划、输出一份完整的“daemon 本地控制面专项 TODO 文档”，要求：

1. 先评估该详细设计是否足以支撑接口级、函数级、数据结构级原子任务拆分。
2. 进行完整的需求确认+任务拆分+任务编排+TODO生成
3. 从详细设计中提取可落盘对象：接口、函数、数据结构、配置模型、错误码、测试点、CMake/注册点。
4. 建立 Design -> TODO 映射，确保每个 TODO 都有明确来源依据。
5. 先设计完整详细的工程实施计划
6. 按最小原子原则拆分任务，默认一项任务只解决一个主目标。
7. 每个 TODO 必须包含代码目标、测试目标、验收命令三件套。
8. 显式输出阻塞项、解阻条件、风险与回退策略。
9. 若设计证据不足，必须把“补设计”作为前置阻塞项，而不是伪造实现任务。

### 4.5 Hard Constraints

1. 不允许改写已冻结 ADR 结论。
2. 不允许越过子系统边界扩张到无关模块。
3. 不允许把纯讨论类事项写成 Done-ready 的 Build 任务。
4. 不允许生成没有代码目标、测试目标、验收命令的任务。
5. 不允许臆造不存在的函数名、字段名、文件路径、测试命令。
6. 若无法细化到函数或数据结构级，必须说明无法细化的证据缺口。
7. 每个任务完成判定必须可二值判定，禁止使用“基本完成”“初步支持”等模糊表述。
8. 若存在 breaking change 风险，必须标记评审门禁，不得在 TODO 中默认推进。
9. 任务拆分必须保持最小原子化；同一任务中不得混入两个以上主实现目标。
10. 所有任务都必须可追溯到详细设计章节、现有代码现状或相关约束文档。
11. 任务拆分时需要考虑Copilot模型能力和上下文容量，避免单次任务过大导致输出失败或者降质输出。

### 4.6 粒度可行性评估规则

在生成 TODO 前，必须先对输入文档做粒度评估，并输出结论：

#### 粒度等级定义

1. L3：函数/方法级
   - 详细设计已明确接口名、方法语义、输入输出、错误语义、调用关系。
2. L2：数据结构/接口级
   - 已明确对象模型、字段、生命周期、依赖方向，但单个函数语义尚未完全展开。
3. L1：组件/类级
   - 已明确组件职责、边界、输入输出，但未明确到接口字段或方法集合。
4. L0：仅设计补完级
   - 当前文档只到概念或职责层，无法安全生成实现 TODO。

#### 允许细化到函数/数据结构级的必要条件

至少满足以下多数条件，才允许输出 L3/L2 级任务：

1. 存在明确的核心接口清单。
2. 存在核心对象或数据结构字段定义。
3. 存在主流程与异常流程说明。
4. 存在错误语义或返回码约束。
5. 存在建议的目录、头文件、实现文件或测试位置。
6. 存在可验证的测试出口或验收路径。

#### 不满足时的处理规则

1. 若只具备组件职责，不具备接口和字段细节，只能输出 L1 级 TODO。
2. 若接口存在但字段、错误语义、测试出口缺失，只能输出 L2 级 TODO，并附“补齐接口/对象设计”前置任务。
3. 若连边界和对象都不稳定，必须输出 Blocked，并给出最小补设计清单。

### 4.7 Required Workflow

#### Step 1：约束与边界抽取

1. 提取子系统职责、层级位置、上下游依赖方向。
2. 提取 ADR、蓝图、工程规范中的 Must / Should / Must-Not 约束。
3. 提取详细设计中的非职责边界、禁止事项、兼容性约束。

输出：约束清单。

#### Step 2：详细设计可执行性扫描

1. 提取子组件清单。
2. 提取核心对象、接口、配置项、错误码、主流程、异常流程。
3. 提取可落盘位置：include/src/tests/CMake/注册点。
4. 对每个子组件做粒度可行性分级（L3/L2/L1/L0）。

输出：粒度可行性评估表。

#### Step 3：Design -> TODO 映射建模

将详细设计中的关键结论映射到 TODO 任务源，至少覆盖：

1. 接口定义类任务
2. 数据结构定义类任务
3. 生命周期与初始化类任务
4. 适配器/桥接类任务
5. 异常与错误处理类任务
6. 配置与 Profile 裁剪类任务
7. 测试与门禁类任务
8. 文档/交付证据回写类任务

输出：Design -> TODO 映射表。

#### Step 4：原子任务拆解

按最小原子原则拆解 TODO，每个任务必须只包含一个主目标，并优先细化到：

1. 单个接口
2. 单个数据结构
3. 单个构造/初始化流程
4. 单个错误路径
5. 单个测试组或单类验证入口

每个任务必须填写：

1. 任务 ID
2. 状态
3. 任务标题（动词开头）
4. 来源依据
5. 设计锚点
6. 粒度等级
7. 代码目标
8. 目标函数/接口/数据结构
9. 测试目标
10. 验收命令
11. 前置依赖
12. 阻塞项
13. 解阻条件
14. 交付物
15. 完成判定

#### Step 5：顺序与门禁编排

1. 给出执行顺序。
2. 标注串行依赖与可并行段。
3. 标注必须先过的门禁：接口冻结、测试注册、构建通过、集成验证。

输出：执行顺序建议与门禁表。

#### Step 6：可行性结论

1. 判断该专项 TODO 是否已经可以直接进入执行。
2. 若可以，说明当前可落到的最细粒度。
3. 若不可以，说明是哪些设计缺口阻止了接口/函数级拆分。

输出：可行性结论与后续建议。

### 4.8 固定输出格式

请严格按以下结构输出最终专项 TODO 文档：

1. 文档头
2. 子系统目标与范围
3. 输入依据与约束清单
4. 粒度可行性评估
5. Design -> TODO 映射表
6. 原子任务清单
7. 执行顺序建议
8. 阻塞项与解阻条件
9. 验收与质量门
10. 风险与回退策略
11. 可行性结论

### 4.9 输出风格要求

1. 任务标题使用工程动作词：新增、补齐、定义、实现、注册、校验、收敛、接线、验证。
2. 优先使用表格表达映射、粒度评估、任务拆解、阻塞项和质量门。
3. 每个结论必须能回溯到输入文档或代码现状。
4. 若某项只能拆到类级或组件级，必须在备注中写清不能继续细化的原因。

## 5. 专项 TODO 文档字段规范

### 5.1 粒度可行性评估表字段

| 字段 | 说明 |
|---|---|
| 设计对象 | 子组件、接口族或对象模型 |
| 设计锚点 | 详细设计章节或表格 |
| 当前粒度等级 | L3/L2/L1/L0 |
| 已具备证据 | 接口/字段/流程/错误码/文件路径/测试出口 |
| 缺失证据 | 缺少什么导致不能继续细化 |
| TODO 拆解策略 | 直接拆任务 / 先补设计 / 标记 Blocked |

### 5.2 Design -> TODO 映射表字段

| 字段 | 说明 |
|---|---|
| Design 项 | 详细设计中的结论或对象 |
| 设计锚点 | 章节、表格、接口列表、流程段落 |
| TODO 类型 | 接口 / 数据结构 / 流程 / 测试 / 配置 / 门禁 |
| 对应任务 ID | 一个或多个专项 TODO 任务 |
| 映射说明 | 为什么这样拆 |

### 5.3 原子任务表字段

| 字段 | 说明 |
|---|---|
| ID | 任务唯一编号，例如 INF-TODO-001 |
| 状态 | Not Started / In Progress / In Review / Blocked / Done |
| 任务 | 动词开头，描述单一主目标 |
| 来源依据 | 详细设计、架构、ADR、代码现状 |
| 设计锚点 | 对应章节、对象、接口、流程 |
| 粒度等级 | L3/L2/L1/L0 |
| 代码目标 | 目录、文件、类、函数、数据结构、注册点 |
| 测试目标 | unit / integration / contract / failure injection |
| 验收命令 | 可直接执行的构建、测试、验证命令 |
| 前置依赖 | 必须先完成的任务或输入 |
| 阻塞项 | 当前无法推进的具体原因 |
| 解阻条件 | 最小可执行解阻动作 |
| 交付物 | 代码、测试、文档、记录 |
| 完成判定 | 可二值判定的完成条件 |

### 5.4 原子任务拆分规则

每个任务必须遵守以下规则：

1. 只能有一个主代码目标。
2. 只能围绕一个接口、一个数据结构、一个初始化路径或一个测试入口展开。
3. 若同一任务同时涉及“接口定义 + 3 个实现类 + 2 类测试”，说明任务过大，必须继续拆分。
4. 若同一任务无法给出明确验收命令，说明尚未达到可执行粒度。
5. 若同一任务只改文档、不改代码和测试，不得记为 Build-ready 原子任务。

## 6. 建议附加检查项

生成 TODO 后，建议执行以下复核：

1. 是否每个任务都能追溯到详细设计中的明确锚点？
2. 是否每个任务都具备代码目标、测试目标、验收命令三件套？
3. 是否存在无法验证的“空任务”或“讨论任务”？
4. 是否把接口冻结、测试发现性、CMake 注册、集成验证单独拆出？
5. 是否把设计缺口误伪装成实现任务？
6. 是否存在明显可以继续拆细但尚未拆细的任务？

---

## 7. 质量门

在接受输出前，必须逐项回答以下问题：

1. 该专项 TODO 是否已经给出 Design -> TODO 映射，而不是只列任务标题？
2. 该专项 TODO 是否已经明确当前最细可达到的粒度等级？
3. 是否所有任务都满足“代码目标 + 测试目标 + 验收命令”三件套？
4. 是否所有 Blocked 项都带有明确证据和解阻条件？
5. 是否所有任务都具备可二值判定的完成标准？
6. 是否避免了跨子系统范围扩张？
7. 若用户要求函数/数据结构级原子任务，输出是否真正落到了这些对象，而不是停留在泛化的组件名称？

若任一问题回答为“否”，则该专项 TODO 不应进入执行阶段。

## 8. 可行性评估结论模板

执行代理在生成结果末尾必须补充以下结论：

1. 结论：可直接生成函数/数据结构级专项 TODO / 只能生成接口级专项 TODO / 仅能生成组件级 TODO / 当前应先补详细设计。
2. 原因：列出支撑该结论的 3-5 条证据。
3. 当前最小可执行粒度：函数 / 接口 / 数据结构 / 类 / 组件。
4. 若未达到函数级：还缺哪些设计信息。
5. 下一步建议：先补设计、先补测试出口、先补文件落点、或直接进入专项 TODO 执行。

## 9. 示例使用建议

对于类似 infra 这类已有“子组件清单、核心对象、核心接口语义、主异常流程、错误码域、测试门禁建议”的详细设计文档，通常可以：

1. 对统一门面、核心对象、错误码、配置模型拆到接口级和数据结构级。
2. 对已有明确方法语义的接口进一步拆到函数级。
3. 对尚未展开内部算法和线程模型的子域实现，先停在接口级或类级。

这意味着“原子任务到具体函数接口/数据结构”是有条件可行，不是无条件可行。判断依据必须来自详细设计是否已经给出足够的接口和对象证据。
