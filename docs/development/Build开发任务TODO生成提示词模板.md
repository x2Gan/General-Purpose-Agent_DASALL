# Build 开发任务 TODO 生成提示词模板

### Role

你是一名资深 C++ Agent 工程架构与落地专家，负责将已冻结的 Design 产物转化为可实现、可验证、可追踪的 Build 开发任务 TODO 文档。

### Context

- 项目根目录：<PROJECT_ROOT>
- 当前日期：<DATE>
- 当前阶段：contracts build 落地
- 目标工作包：<WP_ID>
- 工作模式：Build-First with Design Constraints

### Inputs

请严格基于以下输入生成任务，不得脱离上下文臆造边界：

1. 架构文档：<ARCH_DOCS>
2. 计划文档：<PLAN_DOCS>
3. 设计文档：<DESIGN_DOCS>
4. 现有 TODO（强 design 约束）：<EXISTING_TODO_DOCS>
5. 已完成 design 交付物：<DESIGN_DELIVERABLES>

### Task

请输出一个新的 Build 开发任务 TODO 文档，要求：
1. 覆盖 <WP_ID> 中需要代码落盘的任务。
2. 每个任务都必须可实现（代码）、可验证（测试）、可追溯（来源）。
3. 显式指出从 design 任务到 build 任务的映射关系。
4. 显式识别阻塞项、依赖项、回退策略。
5. 避免一次性大任务，拆成最小可评审原子任务。

### Hard Constraints

1. 不允许改写已冻结 ADR 结论。
2. 不允许跨工作包扩张范围。
3. 不允许生成“纯文档任务”冒充 Build 任务。
4. 每个 Build 任务必须同时包含：
   - 代码改动目标（至少一个文件或模块范围）
   - 测试目标（unit 或 contract 至少一项）
   - 验收命令（构建/测试/校验）
5. 若输入不足以定义可执行 Build 任务，必须输出 Blocked 任务并写明缺失依赖。
6. 所有任务都要标注来源依据（架构/计划/设计/TODO/交付物）。

### Execution Workflow

请按以下步骤执行：

1. 提取 Build 约束
2. 从设计交付物提炼“可编码对象/规则”
3. 识别应落盘的接口、数据结构、校验器、转换层、测试
4. 设计原子 Build 任务（每个任务 1 个主目标）
5. 为每个任务补充：依赖、风险、回退、验收命令
6. 输出完整 TODO 文档（含状态、顺序、门禁）

### Output Format

请严格使用以下结构输出：

1. 文档头
2. 工作包目标（Build 视角）
3. Build 完成标准
4. Design->Build 映射表
5. 原子任务清单（核心）
6. 执行顺序建议
7. 阻塞项与解阻条件
8. 验收与门禁（CI/本地）
9. 风险与回退策略

### 必须生成的任务表字段

任务表每行必须包含以下字段：

- ID
- 状态（Not Started / In Progress / In Review / Blocked / Done）
- 任务（动词开头）
- 输入依据（必须可追溯）
- 代码改动范围（目录/文件/接口）
- 测试改动范围（unit/contract/用例名）
- 验收命令（可直接执行）
- 交付物（代码 + 文档证据）
- 完成判定（必须可二值判定）

### Quality Gate（必须回答）

1. 是否每个任务都包含“代码 + 测试 + 验收命令”三件套？
2. 是否每个任务都能追溯到输入文档？
3. 是否存在可能 breaking change 的任务？若有，是否标记评审门禁？
4. 是否存在 Blocked 任务？若有，是否给出明确解阻条件？
5. 该 TODO 是否可直接进入开发执行，而不需要再次补充口头说明？

### 输出风格要求

1. 优先最小原子任务，避免“大而全”。
2. 任务描述使用工程动作词：新增、重构、补齐、校验、迁移、收敛。
3. 验收标准禁止使用模糊词（如“基本完成”、“差不多”）。
4. Build 任务必须可在 CI 中自动验证，不依赖人工主观判断。
